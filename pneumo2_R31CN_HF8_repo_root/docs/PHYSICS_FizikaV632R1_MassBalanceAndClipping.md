# FizikaV632R1 — Массовый баланс, декомпозиция в/из атмосферы и клиппинги

Этот документ описывает **новую диагностическую надстройку** в `model_pneumo_v9_doublewishbone_camozzi.py`.

Цель: если вариант «плохой», модель должна считать его достаточно далеко, чтобы было видно *почему плохо*:
- где «утекает» масса (в каких элементах идут потоки в/из атмосферы),
- есть ли численная деградация (клиппинги),
- есть ли дрейф (нарушение интегрального баланса).

> Схема пневмосистемы не меняется. Добавлены только аудиты/счётчики.

---

## 1) Временные ряды баланса массы

В `df_main` добавлены столбцы:

- `масса_газа_системы_кг` — фактическая суммарная масса газа в системе (все узлы кроме атмосферы).
- `масса_газа_баланс_кг` — ожидаемая масса по интегральному балансу с атмосферой:

`M_bal(t) = M0_gas + (M_from_atm(t) - M_to_atm(t))`

- `ошибка_массы_газа_кг = масса_газа_системы_кг - масса_газа_баланс_кг`
- `ошибка_массы_газа_отн` — относительная ошибка.

Если всё «хорошо»:
- `ошибка_массы_газа_кг` должна быть около 0 на всём интервале.
- медленный «ползущий» дрейф обычно означает:
  - слишком большой шаг интегрирования,
  - слишком жёсткие нелинейности + недостаточное дробление шага,
  - численные клиппинги состояния.

---

## 2) Декомпозиция массы в/из атмосферы по каждому сопротивлению

В `df_energy_edges` добавлены столбцы:

- `масса_в_атмосферу_кг`
- `масса_из_атмосферы_кг`
- `масса_нетто_кг`
- `объём_Нл_в_атмосферу`
- `объём_Нл_из_атмосферы`

Это интегралы по времени, привязанные к каждой «дросселирующей» связи (`diss_edges`).

Практическое использование:
- сортируйте `df_energy_edges` по `масса_в_атмосферу_кг` — получите «главный выпуск»;
- сортируйте по `масса_из_атмосферы_кг` — получите «главный подсос»;
- смотрите `масса_нетто_кг` — чистый вклад элемента в массобмен.

Также эти же столбцы агрегируются в `df_energy_groups` (groupby по `группа`).

---

## 3) Численные клиппинги (накопленные счётчики)

В релизе добавлены накопленные «счётчики коррекций» состояния:

- `клиппинг_масса_кг`, `клиппинг_масса_count`
  - сколько массы «исправили» из-за отрицательных значений `m<0` после шага.
- `клиппинг_u_энергия_Дж`, `клиппинг_u_энергия_abs_Дж`
  - сколько энергии газа было **внесено/убрано** из-за принудительного клипа `u` в `[u_min, u_max]`.
- `клиппинг_Tw_abs_K`
  - насколько «сильно» приходилось клипать температуру стенки.
- `клиппинг_x_abs`
  - насколько «сильно» приходилось клипать открытия клапанов `x` в `[0,1]`.

Смысл:
- В идеале клиппинг должен быть ~0.
- Если клиппинг не нулевой и растёт, это маркер:
  - «слишком жёстко/слишком большой шаг/слишком резкая нелинейность»,
  - либо параметрический набор приводит к физически недопустимым состояниям.

Важно: клиппинг — это не «физика», это **страховка численного метода**.
Поэтому мы его **логируем**, чтобы не скрывать проблему.

## 4) Сводка в df_atm

В `df_atm` добавлены поля:

- `ошибка_массы_конец_кг`
- `ошибка_массы_max_abs_кг`
- `клиппинг_*` (итоговые значения)

Это даёт быстрый «паспорт качества расчёта» одного прогона.

## 5) Где в коде

Файл:
- `pneumo_solver_ui/model_pneumo_v9_doublewishbone_camozzi.py`

Ключевые места:
- Инициализация: блоки `M_to_atm_edge`, `clip_*`.

- Добавление столбцов: формирование `df_energy_edges`, `df_energy_groups`, `df_atm`.
