# RELEASE NOTES — R33 (2026-01-23)

Ниже — изменения, которые сделаны **в ответ на текущие симптомы**: «анимация не меняется/ползунок мечется», «детальный прогон как будто не останавливается», «битые кэши», «точки ≠ кадры».

## 1) Детальный кэш: надёжность и корректные ключи

### 1.1. Атомарное сохранение (tmp → replace)

**Проблема:** при прерывании сессии/перезапуске/ошибке во время записи мог оставаться *частично записанный* `*.pkl.gz`, который потом не читается (`EOFError` / «нет STOP»). Это провоцировало повторные автозапуски детального прогона и «чехарду» в UI.

**Решение в R33:**
- запись идёт в `*.tmp`, затем `os.replace(tmp, target)`
- при ошибке загрузки/чтения файл **карантинится** в `*.bad<timestamp>`, чтобы не падать повторно и не зацикливаться

События логируются:
- `detail_cache_save_error`
- `detail_cache_load_error`

### 1.2. dt и t_end включены в имя файла кэша

**Проблема:** до R33 путь кэша детального прогона учитывал только `(test_name, max_points, want_full)`.  
Если менялись `dt` или `t_end`, кэш мог быть неверно подхвачен (получается эффект «точки ≠ кадры», не тот таймлайн, не то количество шагов).

**Решение в R33:**
- имя файла кэша включает `dt` и `t_end`, например:
  - `.../detail/ay2__dt0p003__t1p2__mp1200__full1.pkl.gz`

**Совместимость:**
- при отсутствии нового файла загрузчик пробует **legacy-имя** (как в R32),
  и при успешной загрузке **мигрирует** кэш в новый формат имени.

## 2) Playhead/анимация: убран «автозапуск» и дефолтный loop

### 2.1. Restore state выключен

**Проблема:** если в localStorage в прошлом сохранилось `playing=true`, то при открытии вкладки анимации мог стартовать проигрыш автоматически (нагрузка GPU/CPU + «ползунок живёт сам»).

**Решение в R33:**
- компонент playhead вызывается с `restore_state=False` → не подтягивает «залипшее» состояние.

### 2.2. Дефолтные настройки на новый набор данных

При смене набора данных (тест/параметры/`dt`/`t_end`/`max_points`/`full`) теперь отправляется команда:
- `idx = 0`
- `playing = false`
- `loop = false`
- `speed = 0.25`

Это снижает шанс увидеть «пинг‑понг» (быстрое достижение конца + цикл на начало).

## 3) Что НЕ делалось в R33 (чтобы было прозрачно)

- Матемодель не менялась в части «машина едет по дороге, а не дорога под машиной» — это отдельный блок задач (см. `docs/11_TODO.md`).
- 3D-путь «по ax/ay» остаётся **опциональным визуальным режимом** (по умолчанию не используется).

---

## 4) Как проверить, что именно починилось

1) Запусти `RUN_WINDOWS.bat`  
2) Посчитай baseline → выбери тест → дождись детального прогона  
3) Перейди на вкладку **Анимация**:
   - проигрыш **не стартует сам**
   - ползунок при Play идёт вправо и **останавливается в конце** (loop по умолчанию выкл.)
4) Если кэш окажется битым — в `workspace/cache/.../detail/` появится `*.bad<timestamp>`, и деталь пересчитается.

---

## 5) Если проблема всё ещё воспроизводится

Собери «диагностический ZIP» в UI и пришли — в нём будут:
- логи событий (в т.ч. `detail_cache_*`, `playhead_reset`)
- кэши (в т.ч. quarantined `*.bad*`)
- конфиги/входные параметры
