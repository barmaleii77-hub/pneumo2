# Release Notes — R27

Дата: 2026‑01‑23

## 1) Критический фикс: «вечный» авто‑пересчёт full‑лога после baseline
### Симптом (как выглядело у пользователя)
- Нажимаем «Рассчитать baseline».
- Дальше (автоматически) стартует детальный прогон.
- UI показывает, что прогон завершён, но вычисления продолжаются, начинает «сыпаться» анимация.

### Реальная первопричина
В некоторых версиях матмодели `simulate()` **мутирует входной словарь параметров** (добавляет новые ключи/кэш/промежуточные значения).

А UI строит ключи кэша и признак «baseline актуален» через **хеш `base_override`**.
Если `base_override` меняется *после* прогона — хеш меняется, и при следующем ререне Streamlit думает:
- «параметры другие» → baseline считается устаревшим;
- «кэш не совпадает» → full‑лог пересчитывается снова.

Если при этом включён Play/частые ререны (анимация/перерисовка) — создаётся ощущение бесконечного расчёта.

### Что сделано в R27
1. Везде, где вызывается `model.simulate()` из UI/worker‑а, теперь передаются **глубокие копии** параметров и теста:
   - `pneumo_solver_ui/pneumo_ui_app.py` → `call_simulate()`
   - `pneumo_solver_ui/opt_worker_v3_margins_energy.py` → `eval_candidate_once()` и `eval_candidate_once_full()`

Это гарантирует: матмодель может мутировать локальную копию, но **не трогает** объект параметров, который использует UI для хеша/кэша.

### Как проверить (чек‑лист)
- Запустить приложение.
- Нажать «Рассчитать baseline».
- Дождаться завершения baseline + авто‑full‑лога.
- Пошевелить UI (тайм‑слайдер, раскрыть/закрыть expander, Play анимации).

Ожидаемо:
- baseline не «сбрасывается» сам по себе;
- full‑лог не пересчитывается по кругу;
- в `logs/ui_combined.log` нет повторяющихся `detail_start` без явного клика/изменения параметров.

## 2) Исправление краша старта (NameError: Optional)
В R26 добавились тип‑аннотации с `Optional[...]`, но забыли импорт.
В R27 добавлено:

## 3) Колёса: добавлен параметр радиуса и он стал участвовать в модели
Пользовательский вопрос: «почему в матмодели не используется диаметр колёс?»

В R27 добавлено:
- параметр `радиус_колеса_м` (и опционально алиас `диаметр_колеса_м`).
- радиус участвует в вычислении статического положения колеса и в формуле проникновения шины:
  - было: `pen = z_road - zw`
  - стало: `pen = z_road - zw + wheel_radius`

Это пока **не “rolling‑envelope” по пространственному профилю**, но:
- радиус теперь реально участвует в уравнениях;
- абсолютные координаты становятся физически интерпретируемее.

## 4) Автоматический full‑лог снова включён по умолчанию
Ранее “лечили” симптомы выключением автомата.
В R27 автомат снова **ON по умолчанию** (чекбокс «Авто‑расчёт при выборе теста»), потому что причина циклов устранена.

## 5) Анимация — только из результатов реального расчёта
Важно:
- в анимации **нет** подмены физики «красивой синусоидой»;
- дорога/крен/тангаж берутся из рассчитанных временных рядов (`df_main`), а не “рисуются” по названию теста.

Если каких‑то рядов в `df_main` нет — компонент не пытается “угадать” их, а показывает нули/пусто.

## 6) Известные ограничения (не закрыто в этом релизе)
- Модель дорожных воздействий в основном задаётся как `z(t)` (эквивалент “дорога под машиной”).
  Требование «машина едет по дороге» (пространственный профиль `z(x)`, скорость, фазовые задержки + влияние радиуса) вынесено в TODO.
- 3D‑вид остаётся экспериментальным; если не видно — используйте 2D.
