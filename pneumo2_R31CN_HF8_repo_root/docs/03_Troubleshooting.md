# Troubleshooting (типовые проблемы и что делать)

Документ обновлён для релиза **R33**.

---

## 1) Baseline «пропал» после refresh / «надо пересчитывать каждый раз»

Причина: при обновлении вкладки Streamlit создаёт **новую сессию**, и `st.session_state` сбрасывается.

Что сделано:
- baseline и карта тестов сохраняются в **дисковый кэш** и подхватываются автоматически, если найден.

Что проверить:
1) Откройте папку: `pneumo_solver_ui/workspace/cache/baseline/`
2) Внутри должен быть каталог с именем вида `<baseHash>_<suiteHash>_<modelTag>_<modelHash>/`
3) В нём должны быть:
   - `baseline_table.csv`
   - `tests_map.json`
   - `base_override.json`
   - `meta.json`

Если папки/файлов нет — baseline для этой конфигурации ещё не считался.  
Если вы распаковали релиз в **новую** папку и потеряли `workspace/` из старой — кэша действительно не будет.

## 2) «После baseline/детального прогона CPU продолжает грузиться» / «кажется, что детальный прогон не останавливается»

### 2.1. Проверьте: не включена ли синхронизация playhead с сервером
В UI есть опция **«Синхронизация с сервером (тяжело!)»**.  
Если её включить, проигрывание (Play) будет делать регулярные round‑trip в Python → rerun → высокая нагрузка.

По умолчанию в R33 она выключена.

### 2.2. (R33) Битый файл кэша детального прогона
Симптомы:
- ошибки чтения `.pkl.gz` (`EOFError`, «pickle: no STOP», и т.п.)
- авто‑детальный прогон может повторяться, если кэш не удаётся прочитать

Что сделано в R33:
- кэш пишется **атомарно** (tmp → replace)
- при ошибке чтения файл автоматически переименовывается в `*.bad<timestamp>` (карантин), после чего считается заново

Что сделать:
1) Откройте `pneumo_solver_ui/workspace/cache/.../detail/`
2) Если видите много файлов `*.bad...` — соберите диагностический ZIP и пришлите.

### 2.3. (R33) dt и t_end теперь входят в кэш-ключ
Если вы меняете `dt` или `t_end`, это **другой** детальный прогон → будет новый файл кэша.  
Это нормально и убирает проблему «подцепился старый лог, точки/время не совпадают».

---

## 3) «Ползунок анимации мечется туда‑сюда» / «анимация стартует сама»

Что сделано в R33:
- playhead не восстанавливает `playing=true` из localStorage (`restore_state=False`)
- на новый набор данных выставляются дефолты: `playing=false`, `loop=false`, `speed=0.25`

Что проверить в UI:
- Loop выключен
- Speed = 0.25 или 0.5 (для мелкого dt)

## 4) «Кракозябры» в консоли BAT

Обычно это кодировка консоли Windows.
- В BAT уже стоит `chcp 65001`
- Если всё равно «кракозябры» — запускать из Windows Terminal или PowerShell, где UTF‑8 стабильнее.

---

## 5) Высокая нагрузка GPU из‑за графиков

Plotly/Streamlit могут использовать WebGL для больших серий точек (особенно >1000).  
Практические меры:
- уменьшить `max_points` для отображения
- закрыть тяжёлые графики и оставить только нужные каналы
- при необходимости — включить SVG‑рендер (это отдельная опция/тюнинг, см. TODO)
