# 04 — Нулевая поза (rel0) и автономная проверка кинематики

## Зачем это сделано

В проекте нужна **однозначная нулевая поза** для всех графиков и анимации:

- дорога — горизонтальная и на нулевом уровне;
- подвеска — в статике;
- штоки цилиндров — около середины хода (цель);
- все отображаемые перемещения должны считаться **относительно этой статики**.

Параллельно добавлена **автономная проверка (sanity-check)**, которая ловит типичные ошибки интеграции:

- перепутанные индексы углов/колёс/штоков;
- неправильная сборка колонок для графиков/анимации;
- рассогласование «абсолютное перемещение» ↔ «относительное перемещение»;
- рассогласование хода штока с кинематикой double-wishbone.

Это уменьшает риск того, что на первых шагах теста «всё движется одинаково» из‑за багов привязки сигналов.

---

## Что именно сделано

### 1) `*_rel0` колонки в логе модели

Файл:
- `pneumo_solver_ui/model_pneumo_v9_mech_doublewishbone_worldroad.py`

После построения `df_main` теперь автоматически добавляются колонки вида:

- `..._м_rel0` — для всех числовых колонок, заканчивающихся на `_м`
- `..._рад_rel0` — для всех числовых колонок, заканчивающихся на `_рад`

То есть в `*_rel0` хранится **значение минус значение в момент `t=0`**.

Это делает нулевую позу однозначной и удобной для:

- графиков;
- экспортов NPZ;
- оффлайн анимации.

Параметр:
- `use_rel0` (bool, по умолчанию `True`) — можно отключить генерацию `*_rel0` (не рекомендуется).

Дополнительно: если входной профиль дороги содержит **постоянный вертикальный оффсет** (все четыре колеса видят одинаковое значение в `t=0`), то этот оффсет автоматически вычитается (это чистая смена нуля и не меняет физику), чтобы выполнялось условие **дорога(0)=0**.

В `df_atm` пишется:
- `road0_offset_m` — какой оффсет был обнаружен/убран.

Параметр:
- `road0_offset_tol_m` (м, по умолчанию `1e-6`) — порог, ниже которого оффсет считается нулевым.

### 2) Автономный sanity-check кинематики

Файл:
- `pneumo_solver_ui/model_pneumo_v9_mech_doublewishbone_worldroad.py`

После расчёта собираются метрики и кладутся в `df_atm` (одна строка), чтобы их было видно в UI/логах:

- `mech_selfcheck_ok` (0/1)
- `mech_selfcheck_err_wheel_frame_m`
- `mech_selfcheck_err_wheel_road_m`
- `mech_selfcheck_err_stroke_c1_m`
- `mech_selfcheck_err_stroke_c2_m`
- `mech_selfcheck_err_rel0_t0_m`
- `mech_selfcheck_road_t0_flatness_m`
- `mech_selfcheck_warn_lr_identical`
- `mech_selfcheck_warn_fr_identical`
- `mech_selfcheck_msg`

Что проверяется:

1. **Колесо ↔ рама**

Проверка идентичности (с допуском):

`(перемещение_колеса - рама_corner_z) == колесо_относительно_рамы`

2. **Колесо ↔ дорога**

`(перемещение_колеса - дорога) == колесо_относительно_дороги`

3. **Ход штока ↔ кинематика double-wishbone**

Берём `delta = колесо_относительно_рамы` и пересчитываем ожидаемый `s_C1/s_C2` теми же формулами (dw2d/legacy, клип/сглаживание).

4. **Плоскость дороги в момент t=0**

Если `road_func(0)` возвращает разные значения для колёс (неплоский старт), это фиксируется как предупреждение/метрика.

5. **Эвристика «подозрительно одинакового движения»**

Если тест содержит заметный `ay`, а левый и правый борт движутся *точно одинаково* (почти до машинного нуля), ставится `warn_lr_identical=1`.
Аналогично для `ax` — `warn_fr_identical=1`.

Параметры:

- `mechanics_selfcheck` (bool, по умолчанию `True`) — включение проверки.
- `mechanics_selfcheck_tol_m` (float, по умолчанию `1e-6`) — допуск для жёстких равенств.
- `mechanics_selfcheck_strict` (bool, по умолчанию `False`) — если `True`, при провале проверок будет `raise ValueError`.

### 3) Графики в UI: по умолчанию показывают `*_rel0`

Файл:
- `pneumo_solver_ui/pneumo_ui_app.py`

Функция `plot_lines()` теперь, если в данных есть колонка `имя_rel0`, автоматически:

- берёт `*_rel0` как источник данных,
- но сохраняет читаемое имя кривой в легенде (без `_rel0`).

Управление:
- ключ `st.session_state['use_rel0_for_plots']` (по умолчанию `True`).

---

### 4) Анимация (mech_anim): штоки тоже показываются от нуля

Файл:
- `pneumo_solver_ui/components/mech_anim/index.html`

Ранее колёса/рама/дорога рисовались относительно `t=0` (там уже был baseline), а **штоки показывались абсолютным значением**.

Теперь в тексте показывается:

- `Δs / s_abs` в мм

Пример:

`ЛП: +5 / 120 мм`

где `+5` — отклонение от статики, `120` — абсолютное положение штока.

---

### 5) Desktop Animator: абсолютная геометрия как закон

Файл:
- `pneumo_solver_ui/desktop_animator/data_bundle.py`

**LAW (ABSOLUTE-ONLY ANIMATION):** Desktop Animator всегда использует **абсолютные** сигналы
из расчётной модели. Никаких скрытых baseline/offset/«нулевой позы» при отрисовке.

Практически это означает:

- `DataBundle.get()` **не подставляет** `name_rel0` автоматически.
- `*_rel0` допускаются как удобные колонки для графиков/анализа.
- Если в Animator нужно rel0 (например, для текста/оверлея), это делается **явно**
  (`get_rel0()` или обращение к `*_rel0` по имени).

---

## Как проверить руками

1) Запусти UI:

```bash
cd pneumo_solver_ui
streamlit run pneumo_ui_app.py
```

2) Выбери модель:

- `pneumo_solver_ui/model_pneumo_v9_mech_doublewishbone_worldroad.py`

3) Запусти любой тест и убедись:

- графики перемещений идут от нуля;
- в логах/df_atm есть `mech_selfcheck_*` метрики;
- в анимации колёса/рама/дорога отображаются в **абсолютной** геометрии модели;
- штоки могут показывать `Δ/abs`, но координаты для отрисовки берутся из абсолютных сигналов.
