# PneumoApp Windows Desktop GUI Spec

Этот документ задаёт project-specific GUI contract для `Пневмоподвески` поверх общего baseline из [17_WINDOWS_DESKTOP_CAD_GUI_CANON.md](./17_WINDOWS_DESKTOP_CAD_GUI_CANON.md).

Это целевой канон именно для нативного настольного приложения Windows:

- не web-first;
- не mobile-first;
- не браузер в окне;
- не упрощённая миграция с потерей функций;
- не декоративная CAD-оболочка без честной инженерной семантики.

Главный продуктовый контракт:

- главный shell удерживает маршрут и рабочие пространства;
- `Desktop Animator`, `Compare Viewer` и `Desktop Mnemo` остаются отдельными специализированными окнами;
- стандартное Windows-поведение окна, title bar, system menu, drag/maximize/snap semantics и second-monitor workflow считаются обязательной частью desktop contract;
- пользователь всегда видит цепочку
  `Исходные данные -> Набор испытаний и сценарии -> Базовый прогон -> Оптимизация -> Анализ -> Анимация -> Диагностика`;
- матрица соответствия `web -> desktop` является release-gate артефактом и не позволяет молча терять функции при переносе;
- `Редактор кольца` является единственным источником сценарной истины;
- `Базовый прогон` и `Оптимизация` имеют явный runtime-contract;
- `Диагностика` является first-class operational surface;
- графика, анимация и схема обязаны быть honest-by-contract.

Кодовые API и вычислительные контракты этим документом не меняются. Меняется user-facing desktop contract, prompt-contract для GUI-задач и критерии проектирования новых экранов.

## А. Ключевые принципы именно для этого проекта

- Главный shell строится как document-first и viewport-first Windows desktop application: в центре всегда рабочая поверхность, а не длинная прокручиваемая форма.
- Главное окно обязано сохранять нормальные Windows windowing patterns: стандартную строку заголовка, system menu, перетаскивание, двойной щелчок для разворачивания, правый щелчок по заголовку и совместимость со Snap Layouts.
- `Исходные данные`, `Набор испытаний`, `Сценарии`, `Базовый прогон`, `Оптимизация`, `Анализ`, `Анимация` и `Диагностика` являются рабочими пространствами, а не набором случайных страниц.
- Матрица сохранения функций `web -> desktop` обязательна и должна обновляться при каждом значимом переносе поведения.
- `Редактор кольца` является единственным источником истины для сценариев; все generated artifacts являются производными представлениями.
- `Базовый прогон` и `Оптимизация` визуально разделены, но живут в одном логическом run-workspace.
- Для `Оптимизации` существует только один активный режим запуска в конкретный момент времени.
- Стек целевых показателей, жёсткие ограничения, активный контур, источник базового прогона и политика его автообновления считаются first-class runtime contract.
- Команда `Собрать диагностику` всегда заметна и доступна из любого рабочего пространства.
- Служебный шум уводится в `Параметры`, `Настройки`, `Инструменты` и `Диагностику`, но не исчезает.
- Любой незавершённый экран, раздел, кнопка или режим честно маркируется `В разработке`.
- Обозначения без названия и величины без единицы измерения запрещены в пользовательском интерфейсе.
- Каждое значимое поле, кнопка, переключатель, график, предупреждение и заголовок столбца имеют короткую всплывающую подсказку и знак вопроса с развёрнутым объяснением, но ни tooltip, ни help-pane не могут быть единственным носителем смысла экрана.
- Для геометрии, дороги, сценариев, подвески, пневматики и диапазонов движения обязательно двустороннее соответствие `число <-> графика`.
- Для каждого крупного input block обязательны парные формы `число или таблица + графика или схема`.
- `Desktop Animator` и все графические previews работают в одном из трёх режимов: `Расчётно подтверждённый`, `По исходным данным`, `Условный по неполным данным`.
- Каждый preview, график, схема и overlay обязаны показывать источник представления, а расчётные графики и overlays дополнительно показывают время построения.
- Нельзя дорисовывать цилиндры, пружины, рычаги, упаковку или дорогу так, будто они подтверждены расчётом, если данных для этого нет.
- WebView может использоваться только как subordinate technology и не имеет права определять главный interaction model продукта.
- Keyboard-first, accessibility, High-DPI, Snap Layouts, multi-monitor и производительность являются частью definition of done.
- В простое приложение не должно бессмысленно будить процессор, а скрытые панели и фоновые поверхности не должны отъедать бюджет у playback.

## Б. Матрица сохранения функциональности при миграции из веб-версии

| Старая функция | Где жила в веб-версии | Где живёт в настольной версии | Сохранена ли полностью | Что улучшено | Что пока `В разработке` | Как пользователь находит её через поиск команд |
|---|---|---|---|---|---|---|
| Подготовка исходных данных | `Главная` и связанные формы в web-shell | `Исходные данные` | Да | Cluster browser, inspector, readiness-summary, графические схемы | Импорт профилей с полуавтоматическим сравнением | `исходные данные`, `параметры машины`, `геометрия` |
| Редактор кольца | `ui_scenario_ring.py` и web-поверхности набора испытаний | `Сценарии и редактор кольца` | Да | План кольца, профиль, шов, preview артефактов одновременно | Отдельный отчёт по всем конфликтам сегментов | `редактор кольца`, `кольцо`, `сценарии` |
| Генерация дороги и манёвров | `scenario generator` в web | `Сценарии и редактор кольца` | Да | Не скрыта, а встроена в главный сценарный surface | Пакетная генерация нескольких дорожных семейств | `генерация дороги`, `генерация сценария`, `манёвр` |
| Набор испытаний | `suite editor` в web | `Набор испытаний` | Да | Таблица испытаний, связи со сценариями, стадиями и приоритетом | Массовое редактирование нескольких строк | `набор испытаний`, `испытания`, `stage` |
| Базовый прогон | `run setup` и `baseline` flow в web | `Базовый прогон` | Да | Источник истины, явная история, автообновление, provenance | Автоматическая сводка drift между baseline snapshots | `базовый прогон`, `baseline` |
| Поэтапная оптимизация | `Optimization` в web | `Оптимизация`, режим `Поэтапный запуск` | Да | Видны стадии, budget, underfill, gate-reasons и live rows | Сводная диаграмма причин отклонения кандидатов | `поэтапный запуск`, `StageRunner`, `оптимизация` |
| Distributed optimization | `Distributed optimization` в web | `Оптимизация`, режим `Распределённая координация` | Да | Один selector, честный queue/runtime view, контракт истории | Отдельная тепловая карта загрузки worker-ов | `распределённая координация`, `Dask`, `Ray` |
| Analysis / Results | `ResultsViewer` и graph pages в web | `Анализ результатов` | Да | Единая поверхность графиков, таблиц, compare и validation | Сохранённые пресеты многографиков | `анализ`, `результаты`, `графики` |
| Validation cockpit | `validation_cockpit_web.py` | `Анализ результатов` как режим проверки | Да | Не отдельный web-cockpit, а контекст выбранного прогона | Режим сравнения нескольких validation snapshots | `валидация`, `качество прогона` |
| Animation cockpit | `animation_cockpit_web.py` | Отдельное окно `Анимация` | Да | Честный viewport-first, inspector, ViewCube, второй монитор | Расширенный overlay событий по времени | `анимация`, `Desktop Animator`, `трёхмерный вид` |
| Compare NPZ / compare runs | `06_CompareRuns.py`, `compare_npz_web.py` | `Сравнение артефактов` + compare mode в `Анализе` | Да | Сравнение из контекста history и results browser | Сохранённые наборы сравнения | `сравнение`, `compare`, `NPZ` |
| Diagnostics / send bundle | `Diagnostics`, `Diagnostics Hub`, `send_results_gui.py` | `Диагностика` + глобальная кнопка `Собрать диагностику` | Да | Bundle contents, latest ZIP, health, autosave on exit and crash | Автоматический разбор типовых сбоев запуска | `собрать диагностику`, `bundle`, `архив диагностики` |
| Мнемосхема | `PneumoScheme Mnemo` в web и follow-surfaces | Отдельное окно `Мнемосхема` | Да | Follow-mode из выбранного результата, второй монитор | Пользовательские pinned-узлы | `мнемосхема`, `пневмосхема`, `mnemo` |

Правило этой матрицы:

- если функция переносится, она получает новый входной путь, но не теряет смысл;
- если она ещё не готова, точка входа сохраняется и маркируется `В разработке`;
- поиск команд обязан приводить пользователя к новой точке входа и пояснять, где функция теперь живёт.

## В. Информационная архитектура

### Главные рабочие пространства

1. `Обзор`
   Главная инженерная сводка текущего проекта.

2. `Исходные данные`
   Master-copy параметров машины, геометрии, пневматики, механики, масс и базовых настроек решателя.

3. `Сценарии и редактор кольца`
   Master-copy сценариев, сегментов кольца, правил шва и derived artifacts.

4. `Набор испытаний`
   Master-copy набора испытаний: включение, приоритет, стадия, шаг интегрирования, время окончания, привязка сценариев.

5. `Базовый прогон`
   Источник истины по baseline snapshot, истории baseline и политике автообновления.

6. `Оптимизация`
   Источник истины для launch contract, стека целевых показателей, жёстких ограничений, history и runtime-monitor.

7. `Анализ результатов`
   Производные представления, сравнение и validation по артефактам.

8. `Анимация`
   Отдельное специализированное окно визуализации и inspection.

9. `Диагностика`
   Operational surface для self-check, bundle, latest ZIP, autosave и explainable diagnostics.

10. `Параметры приложения`
    Редкие пользовательские настройки.

11. `Инструменты`
    Служебные, справочные и вспомогательные профессиональные утилиты.

### Источники истины и производные представления

- `Исходные данные` — единственный источник истины по конструкции и базовым параметрам.
- `Сценарии и редактор кольца` — единственный источник истины по дороге и маршруту движения.
- `Набор испытаний` — единственный источник истины по составу и метаданным испытаний.
- `Базовый прогон` — единственный источник истины по текущему baseline.
- `Оптимизация` — единственный источник истины по runtime contract.
- `Анализ результатов`, `Анимация`, `Диагностика`, `Сравнение артефактов` — только производные представления.

### Что на главном экране `Обзор`

- карточка текущего проекта: имя, путь, время последнего сохранения;
- лента инженерного маршрута;
- карточка активного baseline;
- карточка активного optimization contract;
- последние результаты;
- последний diagnostics bundle;
- health/self-check summary;
- быстрые действия:
  `Открыть исходные данные`,
  `Открыть сценарии`,
  `Запустить базовый прогон`,
  `Открыть оптимизацию`,
  `Открыть анализ`,
  `Открыть анимацию`,
  `Собрать диагностику`.

## Д. Подробное описание главного окна

### Верхняя командная панель

- содержит menu bar и compact command bar;
- содержит переключатель рабочего пространства;
- содержит единый поиск команд, настроек, сущностей и экранов;
- содержит заметную команду `Собрать диагностику`;
- содержит переключатель режима оптимизации;
- содержит индикатор текущего runtime contract:
  `Целевые показатели`,
  `Жёсткие ограничения`,
  `Источник baseline`,
  `Состояние автообновления`.

### Нативное оконное поведение Windows

- используется обычная Windows title bar semantics без поломки системного поведения;
- правый щелчок по заголовку открывает system menu;
- двойной щелчок по заголовку разворачивает или восстанавливает окно;
- перетаскивание за заголовок и resize borders остаются предсказуемыми;
- Snap Layouts, сценарии половины, трети и четверти экрана считаются обязательными layout targets;
- плавающие панели и detached windows корректно живут на втором мониторе и сохраняют DPI-correct rendering при переносе.

### Левая область обзора

- используется для реальной иерархии:
  кластеров данных,
  сегментов кольца,
  испытаний,
  history,
  artifacts,
  diagnostics bundles;
- если иерархии нет, переходит в list/details или table/details mode;
- поддерживает сортировку, фильтрацию, multi-select там, где это оправдано;
- поддерживает вертикальную прокрутку и явный resize splitter.

### Центральная рабочая область

- `Исходные данные`: редактор + схема + preview + readiness summary;
- `Сценарии`: план кольца + профиль + список сегментов + preview generated artifacts;
- `Набор испытаний`: таблица испытаний + timeline/stage view;
- `Базовый прогон`: baseline card + history + compare to previous;
- `Оптимизация`: launch contract + runtime progress + live outputs;
- `Анализ`: графики, таблицы, compare и validation;
- `Анимация`: viewport-first specialized window.

### Правая область свойств и помощи

- контекстно показывает свойства выбранного объекта;
- содержит предупреждения по этому объекту;
- содержит знак вопроса и развёрнутое объяснение;
- содержит provenance блока:
  `Источник значения`,
  `Расчётно подтверждено`,
  `По исходным данным`,
  `Условно по неполным данным`.

### Нижняя строка состояния и хода выполнения

- краткое состояние системы;
- прогресс и фаза длинной операции;
- active mode;
- предупреждения по расчёту;
- ссылка на подробный журнал;
- путь к последнему ZIP diagnostics bundle;
- дублирование progress на taskbar button для долгих операций.

### Почему это desktop GUI, а не web-форма в окне

Это desktop GUI, потому что:

- интерфейс строится вокруг одной рабочей поверхности, а не вертикальной веб-страницы;
- используются dockable, floating и auto-hide panes;
- есть отдельные специализированные окна;
- раскладки и состояние панелей сохраняются между сеансами;
- поддерживаются system menu, Snap Layouts, second monitor и High-DPI;
- navigation строится через menu, command bar, browser, inspector и command search, а не через web-nav и hamburger.

## Е. Правила помощи и подсказок

### Короткая всплывающая подсказка

- отвечает на вопрос `Что это?`;
- содержит одно предложение;
- при необходимости содержит единицу измерения и сочетание клавиш;
- не объясняет всю методику, а только даёт быстрый контекст;
- не может быть единственным способом понять назначение поля или кнопки.

Пример:

- поле `Шаг интегрирования, секунды (dt)`
- tooltip:
  `Определяет временную дискретизацию расчёта; меньший шаг повышает точность и увеличивает время расчёта.`

### Знак вопроса

- находится рядом с заголовком значимого поля, кнопки, графика, таблицы, предупреждения;
- открывает немодальную панель помощи справа или привязанное окно помощи;
- доступен мышью и клавиатурой;
- не перекрывает рабочую поверхность модальным окном, если для этого нет явной причины.

### Развёрнутое описание

Открывается в `Панели помощи` и содержит поля:

- `Что это`;
- `Зачем нужно`;
- `Как влияет на расчёт`;
- `Единицы измерения`;
- `Допустимый диапазон`;
- `Связанные поля`;
- `Где виден результат`;
- `Что подтверждено расчётом и что показано условно`, если элемент связан с графическим preview.

### Примеры

`Кнопка`

- элемент: `Собрать диагностику`
- tooltip:
  `Собирает актуальный архив диагностики и показывает его состав.`
- help:
  что войдёт в архив, где хранится ZIP, когда выполняется autosave, как читать health/self-check.

`Поле`

- элемент: `Высота конца сегмента, миллиметры`
- tooltip:
  `Задаёт продольный профиль на конце сегмента.`
- help:
  влияние на шов кольца, связь с соседними сегментами, диапазон, связь с preview продольного профиля.

`График`

- элемент: `Перемещение колеса, миллиметры`
- tooltip:
  `Показывает вертикальное перемещение колеса по времени для выбранного испытания.`
- help:
  источник данных, оси, фильтры, где ещё используется этот показатель.

`Предупреждение`

- элемент: `Условный вид цилиндра по неполным данным`
- tooltip:
  `Показан только осевой вид без подтверждённой геометрии корпуса и штока.`
- help:
  что отсутствует, чему доверять нельзя, что нужно для полного расчётного вида.

## Ж. Графическое представление всех вводимых данных

### Общие правила графического представления

- каждый крупный input block обязан иметь как минимум числовое или табличное представление и синхронную графическую поверхность;
- каждый графический preview обязан иметь явный маркер:
  `Источник: расчёт`,
  `Источник: исходные данные`,
  `Источник: условное представление по неполным данным`;
- если preview или график построен по solver/export artifact, рядом показывается `Время построения`;
- если preview строится по неполным данным, рядом показываются причина, перечень отсутствующих данных и список скрытых элементов;
- условный preview не имеет права визуально выглядеть как полноценный расчётный результат.

### Исходные данные: геометрия

- вводятся:
  база, миллиметры;
  колея, миллиметры;
  координаты опорных точек, миллиметры;
  ходы, миллиметры;
  углы, градусы.
- обязательный графический вид:
  схема спереди,
  схема сбоку,
  схема сверху,
  при необходимости трёхмерный каркас.
- число и графика связаны в обе стороны.
- расчётный вид:
  `Расчётное представление` после кинематической проверки, с пометками `Источник: расчёт` и `Время построения`.
- условный вид:
  `Предварительный эскиз по введённым данным` или `Условное представление по неполным данным` с перечислением отсутствующих опорных точек и скрытых элементов.

### Исходные данные: пневматика

- вводятся:
  давления, мегапаскали;
  объёмы, литры;
  диаметры магистралей, миллиметры;
  состояния клапанов, логический признак.
- обязательный графический вид:
  схема пневматической сети с ресиверами, магистралями, клапанами и цилиндрами.
- число и графика связаны:
  выбор узла на схеме открывает свойства, изменение параметра обновляет схему.
- расчётный вид:
  давление и поток после прогона, с источником артефакта и временем построения.
- условный вид:
  только topology diagram по введённым данным с пометкой, что динамика потока и отклики клапанов не подтверждены.

### Сценарии и редактор кольца

- вводятся:
  длина сегмента, метры;
  высота конца сегмента, миллиметры;
  поперечный уклон, проценты;
  поворот;
  режим прохождения.
- обязательный графический вид:
  план кольца сверху,
  продольный профиль,
  поперечный профиль,
  карточка шва.
- расчётный вид:
  generated preview после успешной генерации artifacts с указанием времени генерации и состава derived files.
- условный вид:
  эскиз по введённым сегментам до генерации с явным списком того, какие artifacts ещё не получены.

### Набор испытаний

- вводятся:
  шаг интегрирования, секунды;
  время окончания, секунды;
  приоритет;
  стадия;
  признак включения.
- обязательный графический вид:
  timeline испытаний и стадий,
  coverage-map по сценариям.
- расчётный вид:
  run-progress overlay по фактическому выполнению, с указанием active run artifact и времени последнего обновления.
- условный вид:
  план испытаний до запуска, без имитации реального результата.

### Baseline / Optimization

- вводятся:
  режим запуска;
  стек целевых показателей;
  жёсткие ограничения;
  диапазоны параметров;
  настройки runtime.
- обязательный графический вид:
  карта диапазонов параметров,
  панель ограничений,
  stage timeline,
  graph of candidate flow.
- расчётный вид:
  live rows, stage results, runtime stats, источник текущего run dir и время последнего обновления.
- условный вид:
  launch contract preview, где явно видно, что это контракт поиска, а не статистика выполненной оптимизации.

### Подвеска, цилиндры и пружины

- вводятся:
  точки крепления;
  ход;
  длины;
  зазоры;
  режимы настройки пружин.
- обязательный графический вид:
  схема подвески,
  диапазон перемещения,
  проверка пересечений,
  анимированная схема диапазона.
- расчётный вид:
  подтверждённые geometry and motion states с источником расчёта и временем построения.
- условный вид:
  только оси, проверенные точки и допустимый диапазон, без выдуманной механики, с перечнем скрытых элементов.

## З. Редактор кольца и сценариев

- список сегментов расположен слева и содержит:
  номер,
  тип поворота,
  длину,
  высоту конца,
  поперечный уклон,
  режим прохождения,
  состояние.
- нулевой сегмент задаёт начальные и конечные параметры.
- сегменты с первого и далее задают только конечные параметры.
- последний сегмент автоматически замыкается на начало нулевого сегмента.
- поворот задаётся явно:
  `Прямо`,
  `Влево`,
  `Вправо`.
- продольный профиль задаётся через начальную высоту кольца и высоту в конце каждого сегмента.
- поперечный уклон задаётся одним параметром сегмента, а не отдельными высотами левой и правой колеи.
- разгон и торможение считаются режимом прохождения или телеметрией, а не типом дорожного сегмента.
- пользователь одновременно видит:
  список сегментов,
  графический план кольца,
  продольный профиль,
  preview шва,
  свойства выбранного сегмента,
  preview generated files.
- предупреждения по несогласованности обязательны:
  разрыв шва,
  несогласованность скорости,
  резкий скачок уклона,
  неверная длина,
  отсутствие derived artifacts.
- пустой список сценариев считается штатным состоянием и оформляется как `Сценарии пока не созданы`.

## И. Представление подвески, цилиндров и пружин

### Семейства на борту

- передняя пружина первого контура;
- передняя пружина второго контура;
- задняя пружина первого контура;
- задняя пружина второго контура;
- передний цилиндр первого контура;
- передний цилиндр второго контура;
- задний цилиндр первого контура;
- задний цилиндр второго контура.

### Зеркальность по умолчанию

- левый и правый борта по умолчанию зеркальны;
- асимметрия включается только явным флажком
  `Разрешить асимметрию левого и правого борта`.

### Режимы настройки пружин

- `Ручная настройка`;
- `Автоматический подбор для статического положения`.

В автоматическом режиме обязателен preview:

- целевая середина рабочего хода;
- ожидаемая посадка;
- ограничения по смыканию и rebound.

### Схема цилиндров

Показываются:

- точки крепления к раме;
- точки крепления к рычагу;
- ход цилиндра, миллиметры;
- монтажные длины, миллиметры, если подтверждены;
- длина корпуса цилиндра, миллиметры, если подтверждена;
- положение гланда, миллиметры, если подтверждено;
- вылет штока, миллиметры, если подтверждён;
- зазоры и пересечения;
- проверенный диапазон движения.

### Проверка пересечений

- выделяется на схеме и в списке предупреждений;
- связывается с конкретным семейством цилиндра или пружины;
- показывает минимальный зазор, миллиметры;
- имеет direct command `Открыть пересечение на схеме`.

### Честная деградация при неполных данных

- если нет данных о корпусе цилиндра, показываются только ось и точки крепления;
- если нет данных о гланде или штоке, они скрываются;
- если нет подтверждённой spring geometry, показывается только ось установки и диапазон;
- неподтверждённые поля в inspector помечаются как `Не подтверждено` вместо молчаливого заполнения оценочными значениями;
- рядом обязателен banner:
  `Условный вид по неполным данным`;
- banner объясняет:
  что подтверждено,
  что скрыто,
  что нужно для полного расчётного вида.

## К. Базовый прогон, оптимизация и анализ

### Разделение базового прогона и оптимизации

- `Базовый прогон` — отдельная карточка и отдельная часть workspace;
- `Оптимизация` — соседняя часть того же workspace;
- визуальное разделение поддерживается split layout и отдельными заголовками;
- логическая связь сохраняется через baseline source card.

### Переключатель контура оптимизации

- один selector:
  `Рекомендуемый: Поэтапный запуск`;
  `Расширенный: Распределённая координация`.
- одновременно активен только один режим;
- secondary controls другого режима не показываются как равноправные.

### Видимость целевых показателей и ограничений

На экране до запуска и во время выполнения всегда видны:

- стек целевых показателей;
- жёсткие ограничения;
- режим оптимизации;
- источник baseline;
- политика автообновления baseline;
- путь сохранения результатов.

Дополнительно runtime-contract surface показывает:

- время получения baseline snapshot;
- идентичность активного run contract для history и compare;
- статус `Автоматически обновлять базовый прогон`;
- пояснение, какие параметры и ограничения будут реально участвовать в текущем запуске.

### История запусков

Каждая запись history хранит:

- имя прогона;
- активный режим;
- baseline source;
- stack целей;
- hard gate;
- набор испытаний;
- время запуска;
- состояние завершения.

### Сравнение результатов

Поддерживаются сравнения:

- baseline против лучшего;
- прогон против прогона;
- испытание против испытания;
- сценарий против сценария;
- selected segment against selected segment, если источник данных это допускает.

### Графики

Все графики обязаны иметь:

- полные подписи осей;
- единицы измерения;
- источник данных;
- время построения;
- краткое пояснение, что именно сравнивается.

Графики анализа и экспорта используют те же канонические русские подписи и те же единицы измерения, что и поля ввода и карточки свойств.

### Сохранение и обновление базового прогона

- команда `Закрепить как базовый прогон`;
- история baseline snapshots;
- явный флажок
  `Автоматически обновлять базовый прогон`;
- warning при stale baseline;
- compare against previous baseline.

## М. Доступность, клавиатура и масштабирование

### Контрольный список клавиатуры

- `F6` и `Shift+F6` переходят по регионам:
  верхняя панель,
  левый обзор,
  рабочая область,
  свойства и помощь,
  нижняя строка;
- `Ctrl+K` — поиск команд;
- `Ctrl+S` — сохранить;
- `F5` — запустить базовый прогон;
- `Ctrl+F5` — запустить оптимизацию;
- `Shift+F5` — остановить;
- `F7` — собрать диагностику;
- `F8` — открыть анимацию;
- `Ctrl+Tab` — следующее рабочее пространство.

### Доступные имена и роли

- каждая крупная область имеет explicit accessible name;
- каждое поле имеет name, role, value;
- графики и предупреждения имеют screen-reader-readable summaries;
- custom controls используют UI Automation на уровне стандартных Windows controls.

### Контраст и темы

- поддерживаются светлая, тёмная и контрастная темы;
- смысл не кодируется только цветом;
- контраст текста не ниже `4,5:1`;
- focus всегда видим.

### Масштабирование и multi-monitor

- поддерживаются `100 %`, `125 %`, `150 %`, `200 %`;
- режим DPI-awareness: `Per-Monitor V2`;
- для Win32 path `WM_DPICHANGED` использует suggested rectangle от системы;
- панели корректно живут при переносе между мониторами;
- floating panes и detached windows не блюрятся и не ломают layout;
- Snap Layouts Windows поддерживаются в сценариях половины, трети и четверти экрана.

## Н. Состояния, предупреждения и честность графики

### `Расчётно подтверждённый вид`

- маркер состояния:
  `Расчётно подтверждено`;
- источник:
  solver/export artifact;
- рядом показываются `Источник представления` и `Время построения`;
- пользователь может доверять geometry, movement, overlay metrics и packaging diagnostics.

### `Вид по исходным данным`

- маркер состояния:
  `По исходным данным`;
- используется там, где есть authored data, но ещё нет расчётного подтверждения;
- пользователь может доверять topology и введённым размерам, но не должен путать это с расчётным результатом;
- рядом показывается, что динамика, packaging diagnostics и solver-dependent overlays не подтверждены.

### `Условный вид по неполным данным`

- маркер состояния:
  `Условно по неполным данным`;
- рядом показываются:
  причина,
  отсутствующие данные,
  список скрытых элементов,
  пояснение, чему доверять нельзя.
- если элемент partially reconstructed, рядом указывается, какие части основаны на authored data, а какие не показаны вовсе.

### Где и как показываются предупреждения

- banner над рабочей областью;
- карточка предупреждений в правой панели;
- строка состояния для краткого отражения;
- diagnostics surface для подробной расшифровки.

На каждом графике, preview и 3D-view предупреждение сопровождается указанием источника представления и границ доверия.

### Что пользователь должен видеть как недоверенное

- неподтверждённую геометрию цилиндра;
- неподтверждённый шток;
- неподтверждённую spring body geometry;
- условный профиль дороги;
- восстановленный preview по неполным данным;
- любое восстановленное положение, если solver/export contract его не дал.

### Явные запреты

- нельзя растягивать корпус цилиндра по всей pin-to-pin оси;
- нельзя выдумывать корпус, гланду, шток, поршень и frame fillers;
- нельзя маскировать неполный контракт декоративной прозрачностью;
- нельзя подписывать условную графику как расчётную.

- [ ] Двойной щелчок по заголовку, правый щелчок по заголовку, drag и resize работают по стандартным правилам Windows даже при кастомизации верхней области.

- [ ] Пользователь понимает, где редактируется источник истины и где он видит производный preview.

### Обязательное графическое представление вводимых данных

## Т. Историческая линия `PROMPT_V2 + v1…v13 + v37 + v38` и политика продолжения

Текущий канон не вырос из одного архива. Он собирается из последовательности
слоёв, где разные версии имели разный статус:

- `PROMPT_V2` — upstream prompt source до появления машинных архивов;
- `v1` — стартовый machine-readable GUI-spec;
- `v2` — первый detailed reference layer;
- `v3` — текущий active detailed layer;
- `v4` и `v5` — усиленные design-first layers с richer shell, microcopy,
  state, validation и archive-evidence contracts;
- `v6…v11` — implementation-oriented passes: implementation planning,
  bootstrap shell, fill-in, interactions, backend adapters и execution wiring;
- `v12` — preservation and design recovery: возврат проекта из implementation
  ветки обратно в design-first continuation;
- `v13` — специализированный addendum для `WS-RING` и handoff
  `WS-RING -> WS-SUITE`;
- `v37` — predecessor GitHub KB/TZ/spec supplement: consolidated technical
  specification, GUI_SPEC.yaml, workspace/parameter/acceptance matrices,
  repo canon alignment и список gaps, которые должны оставаться открытыми.
- `v38` — current commit-ready successor KB/TZ/spec layer after V37:
  import-ready subtree, refreshed matrices, optimized pipeline and local
  ambiguity audit. Он не является runtime-closure proof.

### Что уточняет `v37` как predecessor

- `TECHNICAL_SPECIFICATION.md` связывает продуктовый GUI-spec с ТЗ для
  native Windows desktop engineering software.
- `GUI_SPEC.yaml` фиксирует successor machine-readable spec connector,
  включая shell target, graphics honesty states и запрет объявлять package
  runtime closure proof.
- `WORKSPACE_CONTRACT_MATRIX.csv` связывает workspace ids с source-of-truth,
  inputs, outputs, editing rules и layout rules.
- `PARAMETER_*` матрицы задают catalog, grouping, pipeline role, relations и
  visibility/editability для параметров.
- `REQUIREMENTS_MATRIX.csv` и `ACCEPTANCE_MATRIX.csv` дают проверяемый слой
  требований и acceptance, который должен связываться с evidence перед
  объявлением функции готовой.
- `REPO_OPEN_GAPS_TO_KEEP_OPEN.csv` и `NON_RUNTIME_CLOSURE_NOTICE.md` являются
  guardrail: открытые gaps нельзя скрывать, переименовывать в закрытые или
  выдавать за runtime-acceptance.

### Правило использования `v37`

- Для проектирования и ревизии GUI читать `v37` после `v38`, когда нужно
  сравнить predecessor supplement with current V38 commit-ready layer.
- Для реализации нельзя переносить положения `v37` в статус `done` без
  runtime evidence, self-check или acceptance artifact.
- При расхождении `v37` с `17/18` приоритет остаётся у human-readable канона,
  а конфликт фиксируется через assumptions/gaps, а не замалчивается.
