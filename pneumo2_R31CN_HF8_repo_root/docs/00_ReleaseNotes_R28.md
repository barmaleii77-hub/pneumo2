# Release Notes — R28 (2026‑01‑23)

## Цель релиза
R28 — «стабилизационный» релиз после R27, направленный на:
1) устранение краша при старте приложения,
2) подавление **бесконечных авто‑перезапусков** детального прогона (full‑лог) при частых `rerun` в Streamlit,
3) возвращение авто‑режима (после baseline → детальный прогон) в рабочее состояние **без выключения автоматики**.

## Что сделано

### 1) Краш при старте: `use_component_anim` не определена
**Проблема:** при заходе в блок анимации переменная `use_component_anim` могла не быть задана, а ниже использовалась в `if use_component_anim:` → `NameError`.

**Решение:** `use_component_anim` теперь вычисляется сразу после выбора движка анимации (selectbox), до любого использования.

### 2) Бесконечный авто‑пересчёт full‑лога после baseline
**Симптомы (со слов пользователя и по логам):**
- после расчёта baseline автоматически стартует детальный прогон;
- индикатор/спиннер говорит «готово», но сервер продолжает «считать»;
- в `ui_combined.log` видно повторяющиеся события `auto_detail_trigger`/`detail_start` для одного и того же теста;
- из‑за этого «чехорда» с анимацией (в т.ч. возможны повторы и сбросы состояния).

**Причина:** Streamlit может делать частые `rerun` не только по кликам пользователя, но и из‑за:
- `st_autorefresh` (fallback‑анимация, мониторинг процессов, прогресс‑панели),
- включённой синхронизации playhead с сервером (server‑sync),
- событий от интерактивных компонентов,
- интеракций с графиками (Plotly selection/zoom и т.п.).

Если при этом full‑лог запускается «автоматически», то каждый `rerun` может снова попадать в ветку «кэша нет → запускаем full‑лог». Даже если кэш в норме, любой сбой/сброс/неуспешное сохранение мог привести к повторному старту.

**Решение в R28:** добавлен **guard** (страж) авто‑детального прогона.
- В `st.session_state` хранится `detail_guard`: `last_key`, `in_progress`, `last_end_ts`, `suppressed`.
- Если для того же `cache_key` уже шёл расчёт — повторный авто‑запуск не делается.
- Если для того же `cache_key` расчёт только что завершился и тут же снова сработал авто‑триггер (типичный rerun‑loop) — повторный авто‑запуск **подавляется** и показывается предупреждение, что нужно проверить автоперерисовки.
- Важный момент: **ручной** запуск «Пересчитать полный лог» не блокируется (он важнее автоматики).

### 3) Автоматический режим возвращён (не выключаем автомат «в качестве решения»)
В R28 сохраняется логика:
- baseline → можно автоматически догружать детальный прогон для выбранного теста,
- при этом защищаемся от «пилы» rerun‑ов.

То есть «автомат» не выключается как workaround — вместо этого он стал устойчивее.

## Известные ограничения / как жить с `rerun`
- Если включить **«Синхронизация playhead с сервером»**, Streamlit будет rerun'ить скрипт при движении playhead (это нормально по механике Streamlit). В этом режиме лучше:
  - держать `send_hz` минимальным,
  - или отключить server‑sync, если цель — просто смотреть анимацию.
- Fallback‑анимация на matplotlib использует `st_autorefresh` при Play → rerun'ы неизбежны. Это «режим совместимости», а не производительный.

## Где смотреть подтверждение
- `logs/ui_combined.log` — события `auto_detail_trigger`, `detail_start/detail_end`, а также новые `detail_autorun_suppressed` (если guard сработал).
