# Контекст проекта пневмоподвески — сводка по всем файлам (v2)

Дата: 2026-01-24

## 1) Что это за проект (в одном абзаце)

Проект — симулятор пневмоподвески «пневматика + механика» с упором на:
- реалистичную (и в перспективе полностью дифференцируемую) физическую модель,
- подбор/оптимизацию параметров (в т.ч. под реальные компоненты Camozzi),
- визуализацию/анимацию (механика + пневмосхема + графики) для валидации,
- логирование и воспроизводимость прогонов.

## 5) Модель (как она устроена в коде R35, крупными мазками)

### 5.1. Механика (rigid body + 4 колеса)
Есть:
- вертикальное перемещение рамы (z),
- крен/тангаж рамы (φ/θ),
- 4 неразрессоренные массы (колеса) по вертикали,
- силы шин, пружин и цилиндров.

Чего пока **не хватает именно для вашей проверки** (как вы формулировали):
- явных каналов логирования **z/v/a** по каждому углу рамы,
- **z/v/a** центра масс (z есть, v/a можно выводить),
- **z/v/a** каждого колеса *относительно рамы* и *относительно дороги*,
- продольной скорости/ускорения (и/или радиуса поворота и a_c в дуге) в тех же логах,
- «один к одному» скорости для визуализации (тайм‑скейл = реальный).

### 5.2. Пневматика (сеть узлов/ребер)
В модели сеть строится функцией `build_network_full()`:
- узлы: атмосфера, Ресивер1/2/3, Аккумулятор, узлы после регуляторов/клапанов,
  плюс камеры цилиндров (Ц1/Ц2 на каждом колесе, БП/ШП), плюс магистрали противокрена.
- элементы: обратные клапаны (VNR), дроссели (SCO), регуляторы («до себя/после себя»), предохранительный клапан.

Сильная сторона: сеть уже **достаточно детальна**, чтобы логировать давления/расходы/состояния элементов.

Слабые места относительно «реального стенда Camozzi»:
- пока в основном используется эквивалентная площадь A (или ISO‑приближение), **без** расходных карт конкретных регуляторов/клапанов;
- температурная вариативность (окружение/газ в объёмах) в базовом сценарии не разведена как параметр испытания.

## 6) Критически важные расхождения/риски, найденные при просмотре всех пакетов

### 6.1. «ANR / Nl/min / ISO 8778» — есть риск несогласованности в разных файлах
В проекте одновременно встречаются разные «нормальные условия» для пересчёта Qn:
- старый вариант: 101325 Па и 0°C,
- вариант ANR: 100000 Па и 20°C (ISO 8778).

В R35:
- базовая модель `model_pneumo_v8_energy_audit_vacuum.py` уже использует ANR,
- но в комплекте лежит `component_passport.json`, который по ряду компонентов содержит **A, рассчитанные по старым условиям** (это видно по числам: A получается ~на 9% больше).

Следствие: если включить опцию «использовать паспорт компонентов», расход/динамика могут «уплыть» относительно каталожных Nl/min.

### 6.2. Дубли/расслоение модели
Есть несколько файлов модели (base / patched / smooth_all) с частично разъехавшимися константами и логикой.
Это опасно для:
- воспроизводимости,
- калибровки,
- дальнейшего «полного дифференцирования».

## 7) Что лежит в `14.01.26resurses.zip` и зачем оно

Этот архив — набор «сателлитов» вокруг ядра (в большинстве случаев — идеи/патчи, которые полезно переносить в ядро аккуратно):

- **ANR alignment pack**: фикс стандартной атмосферы (p0=100 kPa, T0=293.15 K) и переэкспорт `component_passport` с согласованными A/C.
- **PnevmatikaStepTwenty**: шаг про «digitize flow maps» — подготовка расходных карт регуляторов/клапанов по реальным даташитам.
- **Matematika…Audit**: проверка единиц/контрактов, аудит численной устойчивости.
- **SQL metrics**: перенос логирования метрик из jsonl в SQLite (надёжнее для long‑runs, и проще агрегировать).
- **RealizatsiyaOptimizatsii step22**: robust post‑validation (MVaR / worst‑k) и «cost‑aware» выбор.

## 8) Что я бы закрепил как «контекст проекта» (чтобы дальше не терять)

1) **R35 — рабочая база.** Остальные архивы — как источники патчей/идей/истории.
2) **Геометрия: колея 1.0 м, база 1.5 м** — это фикс в default_base и должно оставаться как «требование №0».
3) Для вашей проверки по прототипу самое важное на ближайшем шаге:
   - расширить набор выходных сигналов (z/v/a углы рамы, колёса, ЦТ),
   - сделать анимацию/визуализацию «один к одному» по времени,
   - привести к единому стандарту пересчёт Qn (ANR) и паспорт компонентов,
   - добавить геометрическую параметризацию креплений цилиндров (4×3 параметра по вашим правилам),
   - добавить ограничения по пружинам (преднатяг на отбое > 0, coil‑bind запас на сжатии).

## 9) Сопутствующие заметки про «полностью дифференцируемую модель»
Сейчас модель написана в numpy‑стиле и не является «энд‑ту‑энд differentiable» автоматически.
Чтобы реально получать градиенты по параметрам без конечных разностей, нужно:
- либо переносить модель на JAX/torch с дифф‑ODE‑солвером,
- либо использовать гибрид: дифференцируемая пневматика (JAX) + внешняя механика.

Это не «сегодня», но контекстно важно, чтобы мы не усиливали расслоение файлов.
