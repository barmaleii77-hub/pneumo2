# Release Notes — Obobshchenie R32

Дата: 2026‑01‑23

## Главное

В этом релизе я целенаправленно лечу **две причины “само‑хаоса” после baseline**, из‑за которых у тебя начинались бесконечные пересчёты/кривое поведение вкладок:

1) **Падение UI из‑за `NameError: test_pick is not defined`**  
   Авто‑детальный прогон и “массовые действия” были фактически вынесены из ветки, где гарантированно определён `test_pick` (когда baseline уже есть и список тестов не пуст).  
   Итог: при отсутствии baseline или при пустом списке тестов код всё равно пытался читать `test_pick`, валился и/или запускал лишнюю логику.

2) **Кнопка “собрать диагностику” падала: `name 'platform' is not defined`**  
   В сборщик диагностического ZIP не был импортирован модуль `platform`.

Плюс добавил **логирование переключения режимов “Графики/Потоки/Энерго‑аудит/Анимация”** в `logs/ui_*.log`, чтобы теперь было видно, *какой именно интерактивный элемент* что триггерит.

---

## Что исправлено

### 1) `NameError: test_pick is not defined` (baseline → auto‑detail)

**Симптомы у тебя:**
- UI падал на строке вида `if test_pick != prev_tp: ...`
- после baseline запускался auto‑detail, прогресс “заканчивался”, но ощущение было что “оно продолжает считать”
- часть логики оказывалась в неожиданных ветках/отступах

**Причина:**
- Блок “авто‑детальный прогон” стоял **вне гарантированной области видимости `test_pick`** (и местами “захватывал” следующую логику из‑за неверных отступов).

**Фикс:**
- Блок авто‑детального прогона переписан и **жёстко “привязан” к ветке**, где `test_pick` уже выбран.
- “Массовые действия” (полный лог по всем тестам, экспорт и т.п.) больше не оказываются внутри авто‑триггеров.

### 2) Диагностический ZIP: `platform` не импортирован

**Симптом:**
- “Не удалось собрать ZIP: name 'platform' is not defined”

**Фикс:**
- Добавлен `import platform` (используется для метаданных в диагностическом архиве).

### 3) Стабилизация UI‑режима результатов

- `view_res` теперь хранится стабильным ключом (не “привязан” к `cur_hash/test_pick`), поэтому при пересчёте baseline/детального лога вкладка меньше “скачет” на “Графики”.
- Добавлено событие `view_switch` в `logs/ui_*.log` при переключении режима отображения.

## Что проверить (короткий чек‑лист)

1) Запуск приложения → вкладка baseline  
   - baseline считается один раз и не падает.
2) После baseline (при включенном “Авто‑расчёт при выборе теста”)  
   - детальный прогон для выбранного теста стартует один раз, завершается и **не перезапускается сам**.
3) Кнопка “Собрать диагностику (ZIP)”  
   - архив формируется и скачивается/сохраняется без ошибки про `platform`.
4) Логи: `pneumo_solver_ui/logs/ui_*.log`  
   - должны появляться события `baseline_*`, `detail_*`, `auto_detail_pending*`, а также `view_switch`.
