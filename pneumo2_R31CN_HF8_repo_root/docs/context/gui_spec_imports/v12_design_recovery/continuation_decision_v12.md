# Continuation decision v12

## Зачем создан этот пакет
Этот пакет делает две вещи одновременно:

1. **Сохраняет все уже созданные артефакты** по GUI/Codex-ветке с комментариями.
2. **Возвращает работу в правильную главную ветку**: дизайн и машиночитаемый контракт, а не преждевременную реализацию.

## Линия ветвления

### Основная ветка проектирования
- `PROMPT_V2`
- `V5`
- `V12`

Это текущий канонический маршрут.

### Сохранённая побочная ветка реализации
- `V6`
- `V7`
- `V8`
- `V9`
- `V10`
- `V11`

Эта ветка **не удаляется** и **не объявляется ошибкой как материал**.  
Но она признаётся **вторичной**: использовать её только после явного freeze дизайна и решения перейти к коду.

## Что считается каноном после v12
1. **Базовая каноническая дизайн-опора**: `PROMPT_V2` + `V5`.
2. **Корректирующий слой продолжения**: `V12`.
3. **Кодовые scaffold-ветки**: только как архив reference, не как основной источник проектных решений.

## Как работать дальше
1. Сначала читать `pneumo_gui_codex_spec_v12_design_recovery.json`.
2. Затем смотреть:
   - `workspace_delta_v12.json`
   - `ring_editor_canonical_contract_v12.json`
   - `optimization_control_plane_contract_v12.json`
   - `truthful_graphics_contract_v12.json`
3. Потом сверять с:
   - `PROMPT_V2`
   - `V5`

## Что делать нельзя
- Нельзя снова незаметно сдвигать работу из ветки проектирования в ветку реализации.
- Нельзя ссылаться на старые сгенерированные документы без повторной проверки их фактического содержимого.
- Нельзя считать `V6–V11` новой канонической дизайн-базой.

## Следующая главная design-задача после v12
Дальше основной ход такой:
- продолжать **чистую проектную ветку**;
- углублять канон `редактор кольца -> тестовый набор -> базовый прогон -> оптимизация -> анализ -> анимация -> диагностика`;
- не писать новый код, пока не будет явной команды на переход к implementation-phase.
