## Что нашёл в свежем bundle
Проверил твой последний Windows SEND bundle `8660b718-5356-4807-9e44-62761788192c.zip`.

Там staged optimization падала не потому, что “ничего не считалось”, а потому что **stage runner ложно объявлял timeout**:

- `bundle/meta.json` показывает релиз **R31BJ**;
- `workspace/opt_runs/main/p_78743e860b24/sp.json` заканчивается на:
  - `status = failed_stage_timeout`
  - `stage = stage0_relevance`
  - `stage_budget_sec = 132.0`
  - `stage_elapsed_sec = 198.65`
- при этом worker progress file `s0/o0_progress.json` уже успел записать:
  - `готово_кандидатов = 3`
  - `готово_кандидатов_в_файле = 1`
  - `прошло_сек = 126.24`

То есть worker **уже посчитал частичные кандидаты**, но stage runner убил его как “stalled”, а часть результатов осталась только в progress JSON и не успела долиться в CSV.

Дополнительно в `_proc.err.log` видны `ResourceWarning` по **незакрытым log file handles** и предупреждение, что subprocess всё ещё running. Это отдельный реальный баг housekeeping/runtime.

## Корень бага
Корень здесь был двойной:

### 1) Worker не был deadline-aware внутри параллельного batch
В `opt_worker_v3_margins_energy.py` time-limit проверялся **между batch-ами**, а не во время самого batch.
Если batch уже стартовал и ProcessPool ещё считает futures, worker мог выйти за бюджет стадии и не успеть нормально завершиться.

### 2) Stage runner пользовался слепым hard-timeout
`opt_stage_runner_v1.py` рубил стадию по wall-time, даже если worker ещё делал прогресс.
В результате:
- часть кандидатов уже была вычислена;
- часть строк ещё сидела в буфере;
- runner объявлял `failed_stage_timeout`;
- CSV оставался неполным.

## Что я исправил: **R31BK**
Собрал следующий patch-релиз:

### В worker
- сделал **deadline-aware parallel loop**;
- worker теперь проверяет **time-limit / stop request прямо во время batch execution**, а не только между batch-ами;
- partial rows теперь **принудительно flush-ятся** перед остановкой по времени/stop;
- для staged drain добавил **fast shutdown/termination ProcessPool**, чтобы worker не зависал до завершения всех lingering tasks.

### В stage runner
- timeout перевёл с тупого wall-time на **activity-based watchdog**;
- stage timeout теперь срабатывает только если worker **реально перестал делать прогресс**;
- parent log handles после `Popen()`/`wait()` теперь **закрываются**, чтобы убрать Windows `ResourceWarning` шум.
