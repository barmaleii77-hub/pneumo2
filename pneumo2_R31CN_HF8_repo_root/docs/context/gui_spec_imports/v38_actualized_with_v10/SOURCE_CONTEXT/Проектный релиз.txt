Сделал следующий шаг — **R31CE**. 🔧📦

На этот раз я закрыл не косметику, а реальный разрыв в оптимизационном UI:

## Что исправлено

### Инженерная страница оптимизации снова умеет **стадийный pipeline**
`03_Optimization.py` раньше жила только через distributed coordinator.
Теперь она честно поддерживает **оба** пути запуска:

- `opt_stage_runner_v1.py` — staged pipeline
- `tools/dist_opt_coordinator.py` — distributed Dask / Ray / BoTorch path

### На странице возвращены **stage-critical настройки**
Прямо в UI добавлены и объяснены:

- `opt_use_staged`
- `ui_opt_minutes`
- `ui_jobs`
- `ui_seed_candidates`
- `ui_seed_conditions`
- `ui_flush_every`
- `ui_progress_every_sec`
- `opt_autoupdate_baseline`
- `warmstart_mode`
- `surrogate_samples`
- `surrogate_top_k`
- `stop_pen_stage1`
- `stop_pen_stage2`
- `sort_tests_by_cost`
- `influence_eps_rel`
- `adaptive_influence_eps`
- `stage_policy_mode`

То есть инженерная страница оптимизации больше не теряет staged-логику, которая уже была в основном UI.

### Появилась **live stage observability** прямо на странице
Во время staged run страница теперь показывает:

- текущую стадию
- live row counts по stage CSV
- `sp.json`
- текущую seed/promotion policy
- focus/explore budget
- underfill / gate reason preview

Это как раз закрывает дыру “стадийность есть в пайплайне, но не видна пользователю на экране”.

### Подтянул runtime hardening и сюда
Страница теперь запускает оптимизацию через:

- `console_python_executable(...)`
- `build_optimization_run_dir(...)`
- `staged_progress_path(...)`
- `PNEUMO_WORKSPACE_DIR`

То есть она больше не обходит старые Windows runtime/path fixes и не тащит регрессии обратно.

### Заодно синхронизировал release markers
Обновлены:

- `VERSION.txt`
- `BUILD_INFO_LATEST.txt`
- `release_tag.json`
- `pneumo_solver_ui/release_info.py`
