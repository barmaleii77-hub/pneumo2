Разобрался. Тут было **две разные проблемы**, и я их закрыл в **R31BW** 🔧📦

## Что реально ломалось

### 1) Почему задаёшь `1`, а сохраняется `0`
Это был **не пользовательский промах**, а баг логики нормализации стадий.

В присланном диагностическом архиве у `ring_test_01` видно сразу три вещи:
- в карточке редактора stage стоял **`1`**
- в persisted `df_suite_edit` строка уже лежала со **`стадия = 0`**
- при этом `_meta_stage_original = 1`

То есть stage **не “показался” нулём**, а был **молча переписан** в ноль.  
Подробный разбор я положил сюда: [R31BW stage semantics / settings audit](sandbox:/mnt/data/r31bw_release/R31BW_stage_semantics_and_settings_audit.md)

### 2) Почему это происходило
Корень был в `normalize_suite_stage_numbers()`:

- если среди **enabled** тестов не было стадии `0`
- функция считала suite “старым 1-based”
- и вычитала глобальный bias

То есть валидный явный suite:
- `ring_test_01`, stage `1`

превращался в:
- `ring_test_01`, stage `0`

Вот это и есть твой симптом.

### 3) Почему стадии “мигрировали как хотят”
Потому что в проекте смешивались **два разных смысла** слова “стадия”:

#### A. `стадия` у строки suite
Это **момент первого входа теста** в staged optimization.

#### B. runtime stage plan
Это сами execution tiers:
- `stage0_relevance`
- `stage1_long`
- `stage2_final`

Их семантика **накопительная**, а не независимая:

- тест с `стадия = 0` идёт в `stage0`, `stage1`, `stage2`
- тест с `стадия = 1` **не должен идти в stage0**, а должен впервые войти в `stage1`, потом остаться в `stage2`
- тест с `стадия = 2` идёт только в финальную стадию

Проблема была в том, что explicit `1` до runtime уже превращался в `0`.  
Поэтому stage plan потом честно работал уже **с испорченным suite**.

## Есть ли уже авто-корректировка точности дифференцирования
**Нет, в staged path её сейчас нет.**

Что есть по факту:
- `system_influence_report_v1.py` умеет принимать **фиксированный** `--eps_rel`
- дефолт там сейчас **`1e-2`**
- но `opt_stage_runner_v1.py` этот `--eps_rel` **не прокидывает**
- и в `pneumo_ui_app.py` **нет UI-настройки** ни для `eps_rel`, ни для adaptive epsilon
- да, в некоторых старых autopilot calibration pipelines есть `--epsilon_adaptive`, но staged optimization path этим **не пользуется**

Значит текущий ответ здесь жёсткий:  
**автоматической корректировки точности дифференцирования по стадиям сейчас нет**.

## Что я исправил в R31BW

### 1) Больше нет молчаливого rebasing `1 -> 0`
Теперь:
- explicit non-negative `стадия` считается **авторитетной**
- negative stage по-прежнему clamp в `0`
- missing/NaN stage по-прежнему infer
- но global bias rebasing выключен

### 2) Empty earlier stage больше не валит StageRunner
Если пользователь реально хочет:
- stage0 пустой
- тест впервые с stage1

то runner теперь:
- **не падает**
- **не переписывает suite**
- а **пропускает пустую раннюю стадию** и идёт дальше

### 3) В UI теперь явнее написан смысл stage
Я обновил help/подписи так, чтобы было ясно:
- stage — это **момент первого входа**
- логика **накопительная**
- `stage 1` не должен молча превращаться в `0`
