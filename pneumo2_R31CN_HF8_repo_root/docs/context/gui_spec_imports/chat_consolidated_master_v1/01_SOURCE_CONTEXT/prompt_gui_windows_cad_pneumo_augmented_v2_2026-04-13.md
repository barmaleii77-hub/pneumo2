# Дополненный промт для проектирования настольного графического интерфейса Windows для проекта «Пневмоподвеска»

Ты — ведущий архитектор пользовательского интерфейса для проекта «Пневмоподвеска».

Нужно спроектировать или перепроектировать графический интерфейс инженерного настольного приложения под Windows для:
- расчёта и настройки пневмоподвески;
- подготовки исходных данных;
- подготовки набора испытаний;
- подготовки и редактирования дорожных сценариев;
- базовых прогонов;
- оптимизации;
- анализа результатов;
- диагностики;
- трёхмерной анимации;
- экспорта и сравнения артефактов расчёта.

Это должен быть именно нативный графический интерфейс Windows для настольного приложения.
Не предлагай веб‑интерфейс как основу.
Не предлагай перенос веб‑страниц в окно как целевую архитектуру.
Не предлагай ухудшение или сокращение существующей функциональности при миграции из веб‑версии в настольное приложение.

Работай только на русском языке.
Не пиши абстрактный манифест.
Не давай общих слов вроде «сделать удобно».
Дай конкретные решения по окнам, областям, командам, состояниям, предупреждениям, справке, графическому представлению данных и критериям приёмки.

────────────────────────────────

## 1. Жёсткий контекст проекта

1. Это инженерный инструмент, а не потребительское приложение.
2. Основной рабочий путь должен быть виден пользователю как единая цепочка:
   **Исходные данные → Набор испытаний и сценарии → Базовый прогон → Оптимизация → Анализ → Анимация → Диагностика**.
3. Генерация дороги и сценариев — полноценная часть продукта, а не скрытая служебная утилита.
4. Диагностика — полноценная функция:
   - одна заметная команда **«Собрать диагностику»**;
   - видимые сведения о составе архива диагностики;
   - видимый путь к последнему архиву диагностики;
   - автоматическое сохранение диагностики при штатном выходе и при аварии;
   - видимый самоконтроль состояния приложения.
5. Никаких скрытых «экспертных» или «устаревших» режимов для критичных функций.
6. Любой незавершённый экран, режим, кнопка или раздел должны быть явно помечены **«В разработке»**.
7. Прогресс длительных операций должен быть виден на текущем экране, а не только в журналах, таблицах или файлах.
8. Пользователь должен видеть один активный режим запуска оптимизации, а не две равноправные конкурирующие кнопки запуска.
9. Базовый прогон должен иметь явный источник истины, видимую историю и явную политику автоматического обновления.
10. Стек целевых показателей, жёсткие ограничения и активный режим оптимизации должны быть видимы пользователю как первоклассный контракт выполнения.
11. Если редактор кольца принят как главный редактор сценариев, он должен быть единственным источником сценарной истины, а остальные представления должны быть только производными: предварительный просмотр, диагностика, экспорт, анализ.
12. Пока контракт геометрии, опорных точек и упаковки цилиндров неполный, интерфейс не имеет права дорисовывать правдоподобную механику как будто она подтверждена расчётом.
13. Если для какого‑то графического элемента нет явных значений из солвера или из исходных данных, такой элемент можно показывать только с явным предупреждением о том, что это условная, оценочная или неполная графика.
14. Потеря существующей функциональности при миграции из веб‑версии в настольный интерфейс запрещена.
15. Вся служебная часть должна быть убрана на задний план — в разделы **«Параметры»**, **«Настройки»**, **«Инструменты»**. Исключение: заметная команда **«Собрать диагностику»**.

## 2. Обязательные правила языка, подписей и единиц измерения

1. Не используй лишние англицизмы. Оставляй только те, которые являются официальными именами технологий, файлов, режимов или классов, например: Windows, StageRunner, Dask, Ray, BoTorch, ViewCube.
2. Обозначения без названий запрещены.
   - Нельзя писать только `dt`, `t_end`, `ax`, `ay`, `RMS`, `C1`, `C2` как основную подпись поля.
   - Нужно писать полное русское название, а обозначение — только вторым планом, например: **«Шаг интегрирования, секунды (dt)»**.
3. Числа и значения без единиц измерения запрещены в пользовательском интерфейсе.
   - Каждое поле, ось графика, подпись маркера, легенда, строка таблицы, карточка свойства и подсказка должны содержать единицу измерения или явно говорить, что это безразмерная доля, логический признак, номер стадии или код режима.
   - Примеры правильной подписи: **«Высота рамы, миллиметры»**, **«Время моделирования, секунды»**, **«Давление, мегапаскали»**, **«Доля потери контакта, доля единицы»**.
4. Для каждого показателя должен существовать единый канонический формат подписи во всём приложении.
5. Заголовки графиков, осей, легенд, таблиц и экспортируемых отчётов должны использовать те же полные названия и те же единицы измерения, что и поля ввода.
6. Иконка без текстовой подписи допустима только если рядом есть короткая всплывающая подсказка и отдельная кнопка вопроса с развёрнутым объяснением.

## 3. Базовые правила нативного настольного интерфейса Windows

1. Строй интерфейс как нативное окно Windows с обычной строкой заголовка, кнопками сворачивания, разворачивания и закрытия, корректным перетаскиванием окна, системным меню, привязкой окон и корректной работой в светлой, тёмной и контрастной темах.
2. Уважай системные привязки окон Windows. Окно приложения должно корректно работать при привязке к половине экрана, трети экрана, четверти экрана, а также при раскладке нескольких окон рядом.
3. Главный центр интерфейса — рабочая область документа или модели:
   - трёхмерная сцена;
   - графики и таблицы результатов;
   - редактор сценариев и дороги;
   - схема подвески и пневматики.
4. Вспомогательные области должны быть оформлены как прикрепляемые панели:
   - прикреплённые;
   - плавающие;
   - автоматически скрываемые;
   - переносимые на второй монитор.
5. Область свойств должна быть контекстной, изменяемой по размеру, прикрепляемой и пригодной для второго монитора.
6. Панели должны запоминать положение и состояние между сеансами.
7. Командная поверхность может быть выполнена как:
   - компактная командная панель;
   - гибрид «вкладки плюс панели»;
   - классическая лента команд;
   - сочетание верхней панели и прикрепляемых инструментальных областей.
8. Полная лента команд допустима только если она действительно улучшает обнаружение команд и обучение. Нельзя выбирать её по умолчанию только ради внешней солидности.
9. Обязателен единый поиск команд, настроек, сущностей и экранов.
10. Если в приложении есть иерархия объектов, используй обозреватель дерева. Если иерархии нет, не притворяйся деревом — используй список, таблицу или карточки.
11. Для общесистемных состояний используй встроенную в макет строку сообщений, не перекрывающую рабочую область. Не закрывай важное содержимое плавающими уведомлениями без необходимости.
12. Для длительных операций можно дублировать ход выполнения на кнопке приложения на панели задач Windows, но только как отражение более подробного индикатора внутри окна, а не вместо него.
13. Если строка заголовка настраивается, не ломай поведение перетаскивания, двойного щелчка для разворачивания, правого щелчка для системного меню и системных привязок.
14. Проектируй под режим **Per‑Monitor V2** для масштабирования точек на дюйм. Никакой размытости, растянутых растров и расползания прикрепляемых окон при переносе между мониторами.
15. При смене масштаба точек на дюйм окно должно корректно перестраиваться с использованием рекомендуемого системой прямоугольника пересчёта.

## 4. Обязательные правила помощи, подсказок и объяснимости

1. У каждого элемента интерфейса должна быть **короткая всплывающая подсказка** при наведении указателя мыши.
2. Рядом с каждым значимым элементом интерфейса должен быть **знак вопроса**, открывающий развёрнутое контекстное описание.
3. Короткая всплывающая подсказка отвечает на вопрос **«Что это?»**.
4. Развёрнутое описание по знаку вопроса отвечает на вопросы:
   - что это;
   - зачем это нужно;
   - как это влияет на расчёт;
   - в каких единицах задаётся;
   - какие допустимые диапазоны и ограничения;
   - с какими полями связано;
   - где пользователь увидит результат этого параметра.
5. Подсказки обязательны для:
   - кнопок;
   - переключателей;
   - флажков;
   - полей ввода;
   - выпадающих списков;
   - переключателей режима;
   - иконок;
   - графиков;
   - легенд;
   - заголовков столбцов;
   - меток состояний;
   - предупреждений.
6. Для повторяющихся ячеек таблиц развёрнутое описание допускается давать на уровне заголовка столбца, строки‑шаблона или карточки поля, а не у каждой отдельной ячейки.
7. Всплывающая подсказка должна быть короткой: одно предложение, ключевая единица измерения, при необходимости сочетание клавиш.
8. Развёрнутое описание должно открываться без модального окна, если это возможно: сбоку, в контекстной панели помощи или в привязанном всплывающем окне.
9. Подсказки и описания не должны быть единственным способом понять экран. Они дополняют хорошую структуру, а не заменяют её.
10. Все элементы помощи должны быть доступны с клавиатуры и читаться программами экранного доступа.

## 5. Правила графического представления вводимых данных

1. Для каждого крупного блока исходных данных обязательно должно существовать синхронное графическое представление.
2. Ввод без наглядного представления запрещён там, где данные описывают геометрию, движение, профиль, диапазон, связь объектов или последовательность событий.
3. У каждого крупного блока должно быть минимум две синхронные формы представления:
   - числовая или табличная;
   - графическая.
4. Обязательные графические представления:
   - профиль дороги по длине;
   - план кольца сверху;
   - поперечный профиль дороги;
   - схема подвески спереди;
   - схема подвески сбоку;
   - схема подвески сверху, если это помогает понять точки крепления и кинематику;
   - схема пневматической сети;
   - графический просмотр диапазонов перемещения;
   - анимированная схема подвески, если вводимые данные связаны с диапазонами перемещения, ходом цилиндров, ходом колеса или рабочими пределами рычагов;
   - временная схема испытаний и стадий;
   - графическое представление активных ограничений оптимизации и диапазонов параметров.
5. Если пользователь редактирует сегмент дороги, он должен одновременно видеть:
   - положение сегмента в кольце;
   - продольный профиль;
   - поперечный уклон;
   - тип поворота;
   - длину сегмента;
   - связь с соседними сегментами.
6. Если пользователь редактирует подвеску, он должен одновременно видеть:
   - схему опорных точек;
   - рычаги;
   - точки крепления цилиндров;
   - зоны допустимых перемещений;
   - возможные пересечения;
   - изменение положения при ходе подвески.
7. Если пользователь редактирует пневматическую часть, он должен видеть не только поля, но и наглядную схему магистралей, ресиверов, клапанов, цилиндров и направлений потока.
8. Каждый графический предварительный просмотр должен быть связан с вводом в обе стороны:
   - изменение числа обновляет рисунок;
   - выбор элемента на рисунке подсвечивает соответствующее поле.
9. Если графика расчётная, это должно быть явно подписано как **«Расчётное представление»**.
10. Если графика строится только по исходным данным без расчёта, это должно быть явно подписано как **«Предварительный эскиз по введённым данным»**.
11. Если графика строится по неполным данным, это должно быть явно подписано как **«Условное представление по неполным данным»** с видимым предупреждением о том, что именно отсутствует.
12. Нельзя показывать условную графику так, будто это проверенный расчётный результат.

## 6. Правила честности графики и представления расчёта

1. Любая геометрия, положение, ход, анимация или наложение должны быть отнесены к одному из трёх режимов:
   - **расчётно подтверждённый вид**;
   - **вид по исходным данным**;
   - **условный вид по неполным данным**.
2. Для каждого режима должен быть свой видимый маркер состояния.
3. Если данные от солвера или экспорта отсутствуют, запрещено молча дорисовывать корпус цилиндра, поршень, шток, пружину, положение контактного пятна, геометрию рычагов или упаковку.
4. Если что‑то дорисовано условно, рядом с графикой должны быть:
   - предупреждающая надпись;
   - краткая причина;
   - перечень отсутствующих данных;
   - указание, что можно доверять, а чему доверять нельзя.
5. Для цилиндров запрещено:
   - растягивать корпус по всей оси между шарнирами;
   - придумывать длину корпуса, положение гланда или вылет штока без явного контракта;
   - маскировать неполный контракт декоративной прозрачностью.
6. Пока полный контракт упаковки цилиндров не получен, допускается только честный осевой вид с явным предупреждением.
7. Для дороги, подвески, анимации и графиков всегда показывай источник представления:
   - по солверу;
   - по исходным данным;
   - условно.
8. Любой расчётный график должен иметь подпись источника данных и времени построения.

## 7. Правила миграции из веб‑версии в настольный интерфейс

1. Потеря функциональности запрещена.
2. Перед проектированием новой структуры составь полную матрицу соответствия **«веб‑функция → настольный экран»**.
3. В матрице для каждой функции укажи:
   - старое место в веб‑версии;
   - новое место в настольной версии;
   - сохранена ли семантика;
   - улучшено ли обнаружение;
   - не потеряны ли параметры;
   - не потеряны ли диагностические поверхности;
   - не потеряны ли сочетания клавиш, фильтры, экспорт, история и режимы выполнения.
4. Нельзя молча убирать из настольной версии:
   - настройки оптимизации;
   - генерацию сценариев;
   - сравнение результатов;
   - диагностику;
   - прогресс выполнения;
   - историю запусков;
   - фильтры;
   - просмотр артефактов;
   - переключение режима оптимизации;
   - скрытые ранее в отдельных страницах, но реально рабочие управляющие параметры.
5. Если функция переносится в другое место, пользователь должен получить:
   - перенаправление из старого сценария;
   - нахождение через поиск команд;
   - краткое объяснение **«где это теперь»**.
6. Если функция временно не реализована в настольной версии, нельзя скрывать факт её отсутствия. Нужно:
   - показать пункт входа;
   - пометить его **«В разработке»**;
   - объяснить, что уже работает;
   - объяснить, чего пока нет;
   - предложить безопасный временный путь, если он существует.
7. Главный экран не должен быть беднее веб‑версии по управляемости. Если служебные панели убираются с первого плана, они всё равно должны быть доступны через понятные разделы и поиск команд.
8. Никакая функция не может исчезнуть только потому, что она кажется «слишком технической». Для технических функций меняется место и глубина, но не само существование.

## 8. Информационная архитектура проекта

Нужны рабочие пространства, а не набор случайных страниц.

Обязательный состав:
- **Исходные данные**;
- **Сценарии и редактор кольца**;
- **Набор испытаний**;
- **Базовый прогон**;
- **Оптимизация**;
- **Анализ результатов**;
- **Анимация**;
- **Диагностика**;
- **Параметры приложения**;
- **Инструменты**.

Правила:
1. Основные инженерные задачи должны быть на первом уровне навигации.
2. Служебные и редкие задачи должны быть на втором уровне навигации.
3. Источники истины должны быть отделены от представлений.
4. Пользователь всегда должен понимать:
   - где он редактирует первичные данные;
   - где он видит производный просмотр;
   - где он видит только результат;
   - где он запускает расчёт;
   - где он запускает диагностику.
5. Нельзя дублировать один и тот же источник истины по нескольким экранам без явного указания, какой из них главный.

## 9. Главный экран и главное окно

Главное окно должно иметь пять крупных областей:
- верхняя командная панель;
- левая область навигации и обзора объектов;
- центральная рабочая область;
- правая область свойств и пояснений;
- нижняя строка состояния и хода выполнения.

Требования:
1. Центральная область должна занимать максимум полезной площади.
2. Левая область используется для дерева, списка испытаний, списка сценариев, истории запусков и артефактов.
3. Правая область используется для свойств выбранного объекта, подробной помощи по знаку вопроса и предупреждений по выделенному элементу.
4. Нижняя область используется для:
   - хода выполнения;
   - активного режима;
   - предупреждений по расчёту;
   - кратких состояний системы;
   - перехода к подробному журналу.
5. Верхняя командная панель должна содержать:
   - поиск команд;
   - основные команды текущего рабочего пространства;
   - переключатель рабочего пространства;
   - быстрый переход к диагностике;
   - переключатель режима оптимизации;
   - видимый индикатор текущего контракта целевых показателей и ограничений.
6. Главный экран не должен превращаться в длинную прокручиваемую форму.
7. Сложные параметры должны жить в панелях свойств, карточках и прикрепляемых областях, а не в бесконечной ленте полей.

## 10. Правила работы с командами, поиском и меню

1. Обязателен поиск команд.
2. Поиск команд должен уметь:

   - находить параметры;
   - находить сущности проекта;
   - открывать нужный экран;
   - подсвечивать, где находится команда;
   - учитывать частоту использования;
   - терпеть частичное совпадение и типичные опечатки.
3. Основные команды должны быть доступны не менее чем тремя путями:
   - через верхнюю командную панель;
   - через поиск команд;
   - через контекстное меню или сочетание клавиш там, где это уместно.
4. Контекстные команды должны зависеть от выделенного объекта и текущей задачи.
5. Нельзя прятать критичную команду только в контекстное меню.

## 11. Правила редактора кольца и сценариев

1. Редактор кольца — единственный источник сценарной истины, если выбран этот проектный канон.
2. У нулевого сегмента должны быть:
   - начальные параметры;
   - конечные параметры.
3. У всех остальных сегментов должны быть только конечные параметры, а начало берётся из конца предыдущего сегмента.
4. Последний сегмент должен автоматически замыкаться на начало нулевого сегмента.
5. Поворот должен задаваться явно:
   - прямо;
   - влево;
   - вправо.
6. Продольный профиль должен задаваться через высоту в начале кольца и высоту в конце каждого сегмента.
7. Поперечный уклон должен задаваться одним явным параметром сегмента, а не отдельными высотами левой и правой колеи.
8. Разгон и торможение — это режим прохождения или телеметрия, а не тип геометрического сегмента.
9. Пользователь должен видеть одновременно:
   - список сегментов;
   - графический план кольца;
   - продольный профиль;
   - предупреждения о шве и несогласованности;
   - свойства выбранного сегмента;
   - предварительный просмотр сформированных файлов сценария.
10. Если список сценариев может быть пустым, это должно быть штатным состоянием, а не ошибкой.

## 12. Правила представления подвески, цилиндров и пружин

   - передняя пружина второго контура;

   - задняя пружина второго контура;
   - передний цилиндр первого контура;
   - передний цилиндр второго контура;
   - задний цилиндр первого контура;
   - задний цилиндр второго контура.
2. Левый и правый борта по умолчанию зеркальны. Асимметрия включается только явным режимом.
3. Для пружин должны поддерживаться как минимум два режима:
   - ручная настройка;
   - автоматический подбор для статического положения с серединой рабочего хода.
4. Для цилиндров должны быть видимы:
   - точки крепления к раме;
   - точки крепления к рычагу;
   - ход;
   - длина корпуса, если она подтверждена;
   - монтажные длины, если они подтверждены;
   - упаковочные ограничения;
   - пересечения.
5. Если данные о корпусе, гланде или монтажных длинах не подтверждены, графика цилиндра обязана перейти в честный условный режим с предупреждением.
6. Если ввод связан с диапазоном перемещения, пользователь должен видеть анимированную или хотя бы интерактивную схему, показывающую этот диапазон.

   - спереди;
   - сбоку;
   - сверху;
   - трёхмерно.
8. Выделение элемента на схеме должно выделять соответствующие поля и наоборот.
