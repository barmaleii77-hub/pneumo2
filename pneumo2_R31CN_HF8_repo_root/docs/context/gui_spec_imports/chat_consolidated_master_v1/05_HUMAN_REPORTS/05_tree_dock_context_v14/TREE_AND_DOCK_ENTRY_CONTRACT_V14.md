# V14 — канон прямого входа из дерева и dock/window-контракт

## 1. Базовое правило
Каждая user-facing инженерная поверхность должна открываться:
- **напрямую из левого дерева**;
- через **поиск команд**;
- без промежуточного «центра окон», «launcher-grid» или повторного выбора уже понятного пути.

Это следует из канона shell:
- видимая основная цепочка работы;
- левое дерево как first-class launcher;
- dock-окна и dock-виджеты как first-class shell-элементы.

## 2. Разделение поверхностей

### 2.1. Workspace-first
Экраны, которые являются источниками истины или основными рабочими пространствами, должны открываться:
- в **центральной рабочей области**;
- с привязкой правого inspector/help;
- с сохранением selection и status/progress полосы.

К этой группе относятся:
- Исходные данные проекта
- Редактор кольца
- Набор испытаний
- Базовый прогон
- Оптимизация
- Анализ результатов
- Диагностика

### 2.2. Advanced secondary
Продвинутые и сравнительные поверхности должны быть вторичными по иерархии, но всё равно открываться из дерева напрямую:
- Подробное сравнение результатов
- Мнемосхема
- Инструменты

Они не должны спорить с основным маршрутом за роль "главного входа".

## 3. Канон по окнам V14

### Диагностика
- Роль: first-class operational surface
- Открытие из дерева: да
- Режим по умолчанию: центральное workspace + правый provenance/help + нижний health/status strip
- Отдельное окно: допустимо как top-level только для dedicated second-monitor / support workflow
- Первое действие: «Собрать диагностику»
- Запрещён competing-route: отдельное стартовое окно «Отправка результатов»

### Подробное сравнение результатов
- Роль: advanced compare route
- Открытие из дерева: да
- Режим по умолчанию: центральная advanced-surface из Анализа
- Отдельное окно: допустимо для second monitor
- Первое действие: выбор левого/правого контекстов сравнения
- Запрещён competing-route: спор со встроенным quick-compare за primary compare

### Исходные данные проекта
- Роль: master-copy входов
- Открытие из дерева: да
- Режим: центральное workspace + графический двойник + inspector
- Первое действие: редактирование конкретной группы входов
- Обязательная обратная связь:
  - dirty-state
  - validation-state
  - visual twin refresh
  - выбранные пружины/цилиндры/симметрия/выравнивание как явно видимые состояния

### Набор испытаний
- Роль: consumer сценарного source-of-truth
- Открытие из дерева: да
- Режим: центральная таблица + preview + inspector
- Первое действие: выбор строки испытания
- Обязательная обратная связь:
  - enabled
  - stage
  - stale/context warnings
  - validated snapshot status
- Возврат при конфликте: прямой переход назад в Редактор кольца

## 4. Правило сохранения контекста
При переходе между окнами shell обязан сохранять:
- project context;
- selected entity;
- dirty/validated state;
- left/right compare picks;
- latest diagnostics bundle reference;
- selected suite row / selected segment / selected parameter group.

Возврат без сохранения контекста допустим только при:
- закрытии проекта;
- переключении на другой проект;
- явном сбросе пользователем.

## 5. Правило Dock / Top-level
- Правая panel свойств и помощи — всегда dock-first.
- Нижняя полоса статуса/прогресса — всегда dock-first.
- Сравнение, мнемосхема и отдельные тяжёлые visual surfaces могут быть top-level, но только как advanced route.
- Main workflow не должен требовать открытия floating окон для базовой работы.
