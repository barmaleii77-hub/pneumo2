# V15 — канон state continuity и repair loops

## 1. Базовое правило
Каждое ключевое рабочее пространство обязано поддерживать **непрерывность состояния**.

Пользователь не должен заново собирать в голове:
- какой проект открыт;
- какой run выбран;
- какой сценарий актуален;
- какой baseline активен;
- почему данные stale;
- куда вернуться, чтобы починить проблему.

## 2. Семейства состояний

### 2.1. `dirty`
Есть локальные пользовательские изменения, которые ещё не сохранены как проектный snapshot.

### 2.2. `invalid`
Есть проверка, которая уже показала ошибку в данных или настройках.

### 2.3. `stale`
User-facing представление основано на устаревшем upstream-источнике:
- stale export;
- stale suite snapshot;
- stale diagnostics bundle;
- stale pointer;
- historical vs current mismatch.

### 2.4. `mismatch`
Два контракта или два контекста не совпадают:
- baseline/source mismatch;
- objective-contract mismatch;
- compare mismatch;
- current vs historical mismatch.

### 2.5. `degraded`
Система сознательно работает в ограниченном режиме, потому что truth или артефакты неполны.
Это не маскируется под нормальный режим.

## 3. Канон repair-loop
У каждого проблемного состояния должен быть:
1. **один главный видимый marker**;
2. **один основной repair-action**;
3. **один понятный return-target**;
4. сохранение selection/context при возврате.

## 4. Обязательная когнитивная видимость
Есть состояния, которые пользователь обязан видеть сразу, а не через deep help:

- На угол выбраны две пружины: контур C1 и контур C2.
- Выравнивание пружинами активно.
- Каким способом выполняется выравнивание.
- На сколько миллиметров выполнено выравнивание.
- Включена ли зеркальная симметрия лево/право.
- Согласован ли шов кольца.
- Актуален ли snapshot набора испытаний.
- Какой baseline сейчас активен.
- Какой objective stack и hard gate сейчас действуют.
- Работа идёт по текущему run или по historical run.
- Аниматор работает в solver/source truth или в degraded truth.
- Последний архив диагностики свежий или устарел.

## 5. Правило возврата
Возврат из repair-loop идёт не “домой”, а в:
- то же рабочее пространство;
- тот же выделенный объект;
- тот же scroll/focus;
- тот же compare context / selected run / selected segment,
если это безопасно.

## 6. Что делать нельзя
Нельзя:
- silently reset selection;
- silently clear compare picks;
- silently заменить current context historical context’ом;
- silently считать degraded режим нормальным;
- отправлять пользователя в отдельный “центр окон”, если нужен ремонт в конкретном upstream workspace.
