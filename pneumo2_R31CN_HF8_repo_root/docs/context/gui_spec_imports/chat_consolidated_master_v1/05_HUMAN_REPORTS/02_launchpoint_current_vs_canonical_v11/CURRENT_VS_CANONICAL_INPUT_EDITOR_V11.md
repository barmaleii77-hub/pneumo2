# Редактор исходных данных — current vs canonical pass

## Current, что честно подтверждено
- Есть отдельная точка входа `Редактор исходных данных`.
- Current internal UX не раскрыт; статус: `launchpoint_only`.
- На launcher-уровне из названия неясно, это главный редактор проекта или техническое окно.

## Что пользователь понимает сейчас плохо
- Это master-copy или вспомогательный редактор.
- Какие именно входы редактируются: весь проект, только инженерные параметры или low-level слой.
- Что будет дальше после изменения входов.
- Где графическое подтверждение изменений.

## Canonical role
- First-class top-level workspace.
- Каноническое имя: `Исходные данные проекта`.
- Это единственная editable-поверхность master-copy для base/ranges/modes до перехода дальше по pipeline.
- Все крупные группы входов обязаны иметь графические двойники.

## Каким должен быть первый экран
1. Слева:
   - дерево групп параметров
2. В центре:
   - форма + графический двойник
3. Справа:
   - свойства поля, единицы, помощь, предупреждения
4. Вверху:
   - статус `сохранено / есть несохранённые изменения / snapshot валиден`
5. Явный next step:
   - `Перейти к сценариям`

## Что исправлять в семантике
- `Редактор исходных данных` → `Исходные данные проекта`
- `Параметры` без контекста → конкретные имена групп
- все значения только с единицами измерения
- все derived preview — явно помечены

## Dock/window contract
- direct-open node в дереве;
- правый dock для properties/help;
- графические двойники как отдельные central tabs/panes;
- не прятать за промежуточным tool center.
