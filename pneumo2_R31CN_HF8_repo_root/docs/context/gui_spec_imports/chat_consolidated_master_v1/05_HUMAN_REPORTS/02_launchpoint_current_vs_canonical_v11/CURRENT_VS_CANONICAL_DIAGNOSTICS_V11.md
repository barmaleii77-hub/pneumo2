# GUI диагностики — current vs canonical pass

## Current, что честно подтверждено
- На launcher/home-surface есть отдельная точка входа `GUI диагностики`.
- Рядом существует отдельная точка входа `GUI отправки результатов`.
- Подсистемно diagnostics зрелая: подтверждены `health_report`, `make_send_bundle`, `selfcheck`, `workspace contract`, `latest_send_bundle`.
- Внутренний первый экран окна в current живо не раскрыт; статус: `launchpoint_only + subsystem_known_window_not_opened`.

## Что пользователь понимает сейчас плохо
- Это одно окно или два соседних окна для одной задачи.
- Сначала надо диагностировать или сразу отправлять.
- Есть ли уже архив, он свежий или устарел.
- Что войдёт в архив и что не вошло.
- Собран полный архив или fallback/частичный.

## Canonical role
- First-class top-level окно/рабочее пространство.
- Прямое открытие из левого дерева.
- Один доминирующий маршрут:
  `Собрать диагностику` → `Проверить состав` → `Открыть / Копировать путь` → `Отправить архив`.
- `GUI отправки результатов` не самостоятельный стартовый путь, а вторичное действие после успешной сборки.

## Каким должен быть первый экран
1. Верхняя полоса состояния:
   - Состояние системы
   - Последний архив
   - Время сборки
   - Свежесть
   - Режим `полный / fallback / частичный`
   - Путь к архиву
2. Центральная primary action:
   - `Собрать диагностику`
3. Блок состава архива:
   - найдено
   - отсутствует
   - будет приложено как fallback
4. Блок после сборки:
   - `Открыть архив`
   - `Открыть папку`
   - `Копировать путь`
   - `Отправить архив`

## Что исправлять в семантике
- `GUI диагностики` → `Диагностика`
- `GUI отправки результатов` → `Отправить архив диагностики`
- `Последний bundle` → `Последний архив диагностики`
- `Self-check` → `Самопроверка системы`

## Dock/window contract
- top-level окно или workspace;
- правый dock: подробности self-check/help/provenance;
- нижний dock: messages/triage;
- левый tree node: `Диагностика`;
- direct open из дерева без промежуточного hub.
