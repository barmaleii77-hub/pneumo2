# Канонические внутренние операционные маршруты

## 1. GUI диагностики

### Главный смысл окна
Пользователь приходит сюда не «посмотреть служебщину», а чтобы:
- понять, живо ли приложение;
- собрать самодостаточный архив диагностики;
- проверить, что именно в архив войдёт;
- увидеть путь к последнему архиву и его свежесть;
- при необходимости перейти к отправке уже **после** готового bundle.

### Первый экран
На первом экране должны быть одновременно видны:
- сводный статус здоровья приложения;
- кнопка **«Собрать диагностику»** как primary action;
- состав архива;
- путь к последнему архиву;
- свежесть архива;
- self-check status;
- явные предупреждения по helper runtime / workspace / missing artifacts.

### Primary route
`Открыть Диагностику → прочитать health summary → нажать "Собрать диагностику" → дождаться bundle → проверить состав и путь → при необходимости "Отправить архив диагностики"`.

### Secondary route
Просмотр triage, self-check, helper runtime provenance, open folder, copy path.

### Что не должно быть первым
- «Отправить результаты» как отдельный competing start;
- голые внутренние пути без объяснения;
- длинная форма опций без health summary.

## 2. Подробное сравнение результатов

### Главный смысл окна
Окно не должно конкурировать со встроенным compare.  
Его роль — **подробное сравнение**, когда quick-compare уже недостаточно.

### Первый экран
На первом экране должны быть видны:
- левый и правый контексты сравнения;
- compare mode;
- contract hash / lineage hash;
- mismatch banner, если контексты несовместимы;
- summary delta по ключевым метрикам;
- выбор графиков/таблиц.

### Primary route
`Открыть подробное сравнение из Анализа → выбрать левый и правый контекст → пройти checks → прочитать summary delta → провалиться в графики и таблицы → экспортировать compare report`.

### Что нельзя прятать
- `current / historical / stale`
- mismatch по objective contract
- mismatch по suite snapshot
- mismatch по scenario lineage

## 3. Исходные данные проекта

### Главный смысл окна
Это **master-copy входов**, а не ещё одно обзорное окно.

### Первый экран
На первом экране должны быть видны:
- текущая группа параметров;
- dirty-state;
- validation-state;
- visual twin активной группы;
- primary actions: `Проверить`, `Сохранить`, `Применить`.

### Что пользователь должен увидеть обязательно
- если на угол выбраны **две пружины** — это видно явно;
- если включено **выравнивание пружинами** — это видно явно;
- каким способом выполнено выравнивание — видно явно;
- остаточная разница после выравнивания — видно числом и единицей измерения;
- включён ли зеркальный режим — видно явно.

### Primary route
`Открыть исходные данные → выбрать группу → изменить поле → увидеть обновление visual twin → запустить проверку → сохранить → перейти в Редактор кольца или Набор испытаний`.

## 4. Набор испытаний

### Главный смысл окна
Это **consumer сценариев**, а не второй сценарный редактор.

### Первый экран
На первом экране должны быть видны:
- таблица строк испытаний;
- тип строки;
- стадия первого входа;
- enabled-state;
- stale-link/context-stale indicators;
- primary actions: `Проверить snapshot`, `Сохранить snapshot`, `Перейти к baseline`.

### Primary route
`Открыть набор испытаний → выбрать строку → проверить stage/dt/t_end/sidecars → получить validation → сохранить validated snapshot → перейти к baseline`.

### Критичное правило
Если stale-link идёт от сценария, возврат должен вести в **Редактор кольца**, а не в попытку починить это внутри набора испытаний.
