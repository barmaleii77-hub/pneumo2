## Статус

V10 является report-only слоем для текущей desktop GUI миграции. Он не содержит кода, но уточняет порядок принятия решений для главного shell и стартовой поверхности.

V10 не отменяет V38, V37, базовый закон проекта, реестр параметров и data-contract. Он уточняет пользовательскую иерархию запуска поверх уже действующего desktop-first канона.

## Главный вывод

Стартовый shell пока выглядит как каталог возможностей, а должен читаться как один сильный пользовательский маршрут.

Обычный пользователь после запуска должен сразу видеть ответ на вопрос: "что делать сначала, потом и позже".

## Канонический маршрут стартового shell

На первом уровне должен быть один доминирующий маршрут:

1. Исходные данные.
2. Сценарии / Редактор кольца.
3. Набор испытаний.
4. Базовый прогон.
5. Оптимизация.
6. Анализ.
7. Анимация.
8. Диагностика.

Этот порядок важнее равноправного каталога всех окон.

## Top-level и second-level решения

Top-level должны оставаться:

- Главное окно проекта.
- Редактор исходных данных.
- Редактор кольца / сценариев.
- Центр тестов.
- Базовый прогон.
- Оптимизация.
- Анализ результатов.
- Desktop Animator как специализированное окно после анализа/прогона.
- GUI диагностики как единая диагностическая поверхность.

Не должны конкурировать с основным стартовым маршрутом:

- GUI отправки результатов: второй шаг внутри Diagnostics после собранного архива.
- Compare Viewer: расширенный режим из Анализа, не основной compare-route.
- Desktop Mnemo: advanced инженерная поверхность из Анализа или Инструментов, не первый обязательный шаг.
- Центр desktop-инструментов: service-hub второго слоя, не первичный шаг маршрута.

## Основные проблемы V10

- Слабая иерархия launcher-shell: слишком много равноправных входов.
- Редактор кольца недостаточно доминирует как сценарный source-of-truth.
- Диагностика и отправка результатов выглядят как конкурирующие направления.
- Embedded compare и Compare Viewer выглядят как дубли без понятной иерархии.
- Исходные данные и Центр тестов слабо читаются как обязательные шаги до baseline и optimization.
- Desktop Mnemo и Центр desktop-инструментов показываются слишком рано для обычного пользователя.
- Оптимизация всё ещё требует короткого объяснения рекомендованного режима.

## Требования к дальнейшим GUI-правкам

- Стартовая поверхность должна показывать маршрут, а не просто набор кнопок.
- Редактор кольца должен быть поднят как явный сценарный центр и единственный источник сценариев.
- В Diagnostics должна быть одна заметная команда "Собрать диагностику"; отправка результатов должна жить после неё.
- Встроенное сравнение должно быть основным compare-path; Compare Viewer должен быть расширенным режимом.
- Исходные данные должны быть названы и показаны как основной набор входов проекта.
- Центр тестов должен быть показан как consumer сценариев.
- Desktop Mnemo и Центр desktop-инструментов должны быть во втором слое с коротким объяснением роли.
- Действия с файлом анимации должны вести к просмотру результата в аниматоре, а не к сырому открытию файла через системную ассоциацию.

## Границы доказательности

V10 не открывал живые внутренности Diagnostics, Send Results, Compare Viewer, Desktop Mnemo и Центра desktop-инструментов. Его выводы основаны на подтверждённых launchpoints, предыдущих report-only слоях и проектном каноне.

Нельзя заявлять runtime closure этих окон только на основании V10. Нужны отдельные visual/runtime проверки.
