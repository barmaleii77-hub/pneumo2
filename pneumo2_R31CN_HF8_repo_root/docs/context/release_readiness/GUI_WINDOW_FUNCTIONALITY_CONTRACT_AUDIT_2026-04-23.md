# GUI window functionality and contract compliance audit - 2026-04-23

## Scope And Sources

This audit compares real window functionality against the project GUI contracts and knowledge base. It does not judge by window names. It checks whether each user task has one primary dock/widget route, whether old windows duplicate the same task, and whether the primary dock has the same practical capability as the older specialized window.

Primary sources:

- `docs/00_PROJECT_KNOWLEDGE_BASE.md`: `desktop_main_shell_qt` is the public GUI target; `desktop_gui_spec_shell` is support/dev; the left tree opens dock/widget/window directly; ring editor is the only editable scenario source of truth.
- `docs/18_PNEUMOAPP_WINDOWS_GUI_SPEC.md`: workspaces are real working surfaces, not random pages; active GUI structure follows the Desktop Animator pattern: one `QMainWindow`, `QDockWidget` child panels, movable/floatable/closable docks, direct tree click.
- `docs/context/release_readiness/GUI_CANONICAL_WINDOW_MEMORY_2026-04-23.md`: no mandatory launcher-grid, no center-of-windows before real editors, no second choice after tree click.

## Root Cause

The architectural direction is correct, but the migration cut was incomplete. Work moved the shell toward the contract first: `QMainWindow`, left tree, docks, immediate hosted pages, command surface. However, old standalone windows still contain significant paid/domain functionality. Where a hosted dock only contains a summary/control hub, the user correctly sees a second interface: the "new" shell plus the old real editor.

The failure is not that old windows exist. The failure is exposing old windows as the place where real work still happens while the contract says the left-tree route itself must open a complete dock/widget for that stage.

## Implementation Direction

Do not delete legacy windows before parity. The safe path is:

1. Keep `desktop_main_shell_qt` as the only public GUI target.
2. Keep legacy windows callable only as explicit fallback/support actions.
3. For each domain, move missing legacy functionality into the hosted dock or into a child/floating dock managed by the main shell.
4. After parity is reached for a domain, remove its legacy window from command search and normal launcher/tree surfaces.
5. Continue with rich result QA, because the basic suite/autotest route, full project check, send-material review, run-material review, selected-material review, chart drilldown, compare-contract review, engineering-readiness review, analysis candidates, selected-run pinning, system-influence launch, full-report launch, parameter-staging launch, influence review, compare-influence review, engineering manifest save and analysis-to-animation link are now hosted but advanced multi-run compare/timeline controls still have fallback windows.
