# Autotune точности интегрирования

## Что именно «автотюнится» сейчас

В симуляторе используются два масштаба шага времени:

- `dt` — **шаг логирования** (шаг дискретизации выходных рядов в DataFrame).
- `dt_int` — **внутренний шаг интегратора**, которым реально интегрируется модель внутри каждого `dt`.

Текущая стратегия (во всех актуальных UNIFIED-моделях):

- интегратор **RK2 (Heun, явный метод 2-го порядка)**,
- интегрирование выполняется с **подшагами**, причём `dt_int` выбирается автоматически как минимум
  из нескольких ограничений «на устойчивость/физическую корректность».

### Основные ограничения на внутренний шаг

1) **Верхняя граница**:

- параметр: `макс_шаг_интегрирования_с` (далее `dt_int_max`).

Идея: пользователь задаёт максимально допустимый внутренний шаг,
а реальный `dt_int` никогда его не превышает.

2) **Ограничение по изменению объёма** (аналог CFL для геометрии):

- параметр: `лимит_относит_изменения_объёма_за_шаг` (обычно 0.05 = 5%).
- формально: требование вида `|dV| * dt_int / V <= lim`.

Это критично в режимах, когда объём камеры резко меняется (быстрый ход штока),
потому что давление/энергия газа становятся чувствительны к `dt_int`.

3) **Ограничение по массе / «антивакуум»**:

- шаг ограничивается так, чтобы масса в узлах не уходила «слишком быстро»
  (и/или не пересекала нижний «пол» по массе/давлению).

4) **Защита от зависания**:

- параметр: `макс_число_внутренних_шагов_на_dt`.

Если внутренний шаг стал слишком малым (модель попала в жёсткий режим),
симуляция не должна уходить в бесконечный цикл.

## Как интерпретировать это с точки зрения «точности»

Важно понимать: текущий механизм — это **не полный адаптивный ODE-солвер с контролем локальной ошибки**,
а практичный «автотюн шага» по физическим ограничениям + ограничение `dt_int_max`.

Тем не менее, уменьшение `dt_int_max` (и/или более жёсткие лимиты по объёму/массе)
обычно повышают точность, потому что RK2 имеет порядок 2.

## Автоматическая проверка (pytest)

Добавлен тест-конвергенция, который проверяет, что уменьшение `макс_шаг_интегрирования_с`
даёт заметное улучшение результата (на гладком сценарии без разрывов):

- файл: `tests/test_integrator_autotune_accuracy.py`
- тест: `test_integrator_dt_int_max_convergence_worldroad`

Запуск:

```bash
pytest -q -k integrator_dt_int_max_convergence_worldroad
```

Тест сравнивает решение для `dt_int_max=3e-4` и `dt_int_max=1.5e-4` с более мелким «эталоном» `7.5e-5`
и требует, чтобы средний вариант был лучше грубого (и улучшение было не «на шум»).

## Автономная самопроверка (без pytest)

Чтобы ловить регрессии даже когда пользователь запускает UI без тестов,
добавлена автономная smoke-проверка:

- модуль: `pneumo_solver_ui/tools/integrator_autotune_smoke_check.py`
- запуск:

```bash
python -m pneumo_solver_ui.tools.integrator_autotune_smoke_check
```

Она же подключена в `pneumo_solver_ui/tools/autoselfcheck.py`.
Отключение в окружении:

```bash
PNEUMO_AUTOCHECK_NO_AUTOTUNE=1
```


## Диагностика подшагов интегратора (h-статистика)

Чтобы проще понимать, **почему модель стала жёсткой/медленной** и действительно ли
`макс_шаг_интегрирования_с` применяется как «ручка точности», добавлены диагностические колонки
в `df_main` (в основных механических UNIFIED-моделях):

- `model_pneumo_v9_mech_doublewishbone_worldroad.py` (worldroad),
- `model_pneumo_v9_doublewishbone_camozzi.py` (camozzi),
- `model_pneumo_v9_mech_doublewishbone_r48_reference.py` (r48_reference).

- `интегратор_подшаги_N` — сколько внутренних подшагов было сделано внутри одного внешнего `dt`;
- `интегратор_подшаг_мин_с` — минимальный подшаг `h` за интервал `dt`;
- `интегратор_подшаг_макс_с` — максимальный подшаг `h` за интервал `dt`;
- `интегратор_подшаг_средний_с` — средний подшаг `h` за интервал `dt`.

Особенности:

- эти колонки **не влияют на расчёт** (только логирование);
- для последней строки `df_main` значения могут быть пустыми/NaN (там нет следующего интервала `dt`).

Smoke-check и pytest-тест дополнительно проверяют, что `h_max <= макс_шаг_интегрирования_с` (сейчас — на сценарии worldroad).

## Рекомендации по практическому выбору параметров

- Если нужна **быстрая отладка**: держите `макс_шаг_интегрирования_с` порядка `3e-4 … 5e-4`.
- Если важна **точность давлений/энергии** в жёстких режимах: пробуйте `1e-4 … 2e-4`.
- Если симуляция «затыкается» по лимиту внутренних шагов:
  - либо уменьшайте жёсткость сценария/параметров (слишком сильные перетоки/скачки),
  - либо увеличивайте `макс_число_внутренних_шагов_на_dt`,
  - либо уменьшайте `dt` (шаг логирования), чтобы один макрошаг был короче.

## Следующий уровень: настоящий контроль локальной ошибки (план)

Если потребуется именно **автотюн точности по допуску** (rtol/atol),
правильный путь — внедрить встроенную оценку ошибки (embedded RK):

- иметь два приближения на шаге (например, Heun(2) и Euler(1)),
- оценивать локальную ошибку и принимать/отклонять шаг,
- автоматически выбирать следующий шаг.

Это повысит «математическую строгость» управления точностью, но может быть дорогим на жёстких режимах.
