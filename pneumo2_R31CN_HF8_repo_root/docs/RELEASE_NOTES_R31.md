# RELEASE NOTES — R31 (2026‑01‑23)

Этот релиз сфокусирован на стабилизации **авто‑детального прогона (full‑лог)** и устранении конфликтов с анимацией/Play.

## 1) Исправление: `detail_dt` / `detail_t_end` не определены

**Симптом:** приложение падало с `NameError: name 'detail_dt' is not defined` (и аналогично `detail_t_end`), обычно при попытке автоматически стартовать детальный прогон.

**Причина:** в блоке детального прогона использовались переменные `detail_dt/detail_t_end`, которые не создавались (планировались как UI‑override, но фактически отсутствовали).

**Что сделано:**
- Убраны обращения к `detail_dt/detail_t_end`.
- `dt/t_end` берутся из выбранного теста (`tests_map[test_pick]['dt'/'t_end']`), а если там пусто — используются дефолты `0.01 / 1.0`.
- Логирование `detail_start` перенесено туда, где `dt/t_end` уже гарантированно определены.

## 2) Исправление: `call_simulate()` допускает `dt=None`, `t_end=None`

**Симптом:** в некоторых сценариях (особенно в диагностике/скриптах) появлялась ошибка:
`call_simulate() missing 2 required keyword-only arguments: 'dt' and 't_end'`.

**Что сделано:**
- `call_simulate()` теперь принимает `dt: Optional[float] = None`, `t_end: Optional[float] = None`.
- Если `dt/t_end` не переданы — они подхватываются из `test/params`. Если и там нет — используются дефолты (`0.01 / 1.0`).
- Сохранён механизм совместимости по сигнатуре: лишние kwargs фильтруются, а список отброшенных аргументов логируется событием `call_simulate_dropped_kwargs`.

## 3) Исправление: авто‑детальный прогон больше не стартует на каждом rerun

**Симптом (ключевой):**
- После baseline автоматически начинался full‑лог.
- Индикатор мог показывать «готово», но CPU продолжал грузиться.
- Любое действие в UI (включая Play) могло снова запускать heavy‑расчёт, что приводило к «чехарде» и дерганью ползунка/анимации.

**Причина:** логика была вида:
`if auto_detail_on_select: if cache miss -> run detail`,
а Streamlit делает rerun на любые события (кнопки, компоненты, переключения), поэтому авто‑режим невольно срабатывал повторно.

**Что сделано:**
- Введён **pending‑триггер** в `st.session_state`:
  - `detail_auto_pending` выставляется **только** при:
    - смене выбранного теста, или
    - обновлении baseline (фиксируется `baseline_updated_ts`).
- Детальный прогон запускается только если:
  - пользователь явно нажал кнопку пересчёта, или
  - `auto_trigger == True` (pending соответствует текущему cache_key).
- При входе в блок расчёта pending сразу **сбрасывается** (consume), чтобы исключить повторные авто‑старты при следующих rerun.

## 4) Что это даёт для анимации/Play

- Нажатие Play больше не провоцирует «случайный» старт детального прогона.
- Если full‑лог уже готов — Play работает “чисто” (перемотка по готовым данным).
- Если full‑лог ещё не готов — теперь UI логика не должна уходить в бесконечные расчёты из‑за rerun; детальный прогон запускается только по pending‑триггеру.


- В 3D‑виде дополнительные «траектории» (слалом/поворот/интеграция ускорений) теперь спрятаны за чекбоксом **«3D: доп. траектории (НЕ физика, только визуализация)»** и по умолчанию выключены, чтобы не создавать впечатление «подрисованной физики».

## 5) Известные ограничения / честные допущения

- Диаметр/радиус колёс в текущей матмодели может **не участвовать** в динамике, если входные возмущения дороги задаются сразу в координатах центра колеса (тогда радиус «схлопывается» в константу). Для физически корректного учёта радиуса нужна модель контактного пятна/обкатки препятствий (TODO).
- Анимация должна рисовать **только результаты расчёта**. Если каких‑то колонок в результате нет — анимация показывает нули/не рисует соответствующий слой, но **не** “фейкует физику”.
