# Единый реестр параметров

Этот документ фиксирует **где лежит канонический перечень параметров** и **какие ключи считаются интерфейсными** между модулями.

Он существует, чтобы исключить ситуацию: «в одном месте ожидают одно имя, а приходит другое».

---

## Канонические источники ключей

### 1) Параметры модели (входные)

* **`pneumo_solver_ui/default_base.json`** — канонические ключи и дефолты параметров модели.
* **`pneumo_solver_ui/default_ranges.json`** — диапазоны/ограничения для параметров модели.

Правило:

* Если параметр существует в `default_base.json`, **его имя канонично**.
* Новый параметр **нельзя** вводить “просто в одном файле/модуле”. Нужно обновлять **оба** источника (и документировать).

### 2) Схема набора тестов (suite)

* **`pneumo_solver_ui/default_suite.json`** — канонический пример структуры suite и ключей теста.

Канонические колонки/поля записи теста:

* `id` — стабильный идентификатор теста
* `имя` — имя теста (уникально в рамках suite)
* `тип` — тип теста
* `включен` — включён/выключен

(Доп. поля допускаются, но только если они документированы и используются в рамках контракта.)

### 3) Контракт экспорта результатов (NPZ/Animator)

* **`DATA_CONTRACT_UNIFIED_KEYS.md`** — канонические ключи метаданных `meta.*` и канонические имена сигналов в данных.

Это **не полный** список всех параметров модели, а тот минимум, который критичен для сквозной совместимости UI → расчёт → экспорт → Animator.

* `axay_csv` — путь к CSV манёвра/ускорений (если используется)

## Служебные/производные значения

Служебные величины разрешены **только как производные** от канонических параметров.

Пример:

* `svc__s_world_м` (путь вдоль дороги) может быть вычислен из `скорость_vx_м_с` и `время_с`,
  но должен быть помечен как `SERVICE/DERIVED` и не должен появляться как «сигнал модели».

## R17 stage‑1 — канонические семейства spatial DW source‑data

Эта секция вводит **переходный канон R17** для пространственной подвески double wishbone.

### Главный принцип

- Старые `dw_*` ключи (`dw_lower_arm_len_*`, `dw_upper_pivot_inboard_*` и т.п.) остаются только как **donor-only / offline migration**.
- Runtime‑каноном для новой геометрии становятся **явные hardpoints**, а не выведенные длины/полуразносы.
- Недостающие hardpoints **нельзя восстанавливать** из legacy `dw_*` тихо в solver/animator.

### Новые канонические семейства

#### 1) Hardpoints верхнего/нижнего рычага

Для каждого `рычаг ∈ {верхний_рычаг, нижний_рычаг}`
и `ось ∈ {ось_перед, ось_зад}`:

- `{рычаг}_{ось}_рама_ветвь_перед_{x|y|z}_м`
- `{рычаг}_{ось}_рама_ветвь_зад_{x|y|z}_м`
- `{рычаг}_{ось}_ступица_ветвь_перед_{x|y|z}_м`
- `{рычаг}_{ось}_ступица_ветвь_зад_{x|y|z}_м`

#### 2) Верхние точки цилиндров

Для каждого `Ц ∈ {Ц1, Ц2}` и `ось ∈ {перед, зад}`:

- `верх_{Ц}_{ось}_x_относительно_оси_ступицы_м`
- `верх_{Ц}_{ось}_z_относительно_рамы_м`
- `верх_{Ц}_{ось}_между_ЛП_ПП_м` (для `ось=перед`) / `верх_{Ц}_{ось}_между_ЛЗ_ПЗ_м` (для `ось=зад`)

#### 3) Нижний шарнир штока на рычаге

Для каждого `Ц ∈ {Ц1, Ц2}` и `ось ∈ {перед, зад}`:

- `низ_{Ц}_{ось}_рычаг_крепления` ∈ `{верхний_рычаг, нижний_рычаг}`
- `низ_{Ц}_{ось}_ветвь_трапеции` ∈ `{перед, зад}`
- `низ_{Ц}_{ось}_доля_рычага` ∈ `[0, 1]`

#### 4) Физика цилиндров

Сохраняются как канон:

- `диаметр_поршня_Ц1`, `диаметр_поршня_Ц2`
- `диаметр_штока_Ц1`, `диаметр_штока_Ц2`
- `ход_штока_Ц1_перед_м`, `ход_штока_Ц1_зад_м`
- `ход_штока_Ц2_перед_м`, `ход_штока_Ц2_зад_м`

### Где держать машинный список

Машинный список ключей и валидация лежат в:

- `pneumo_solver_ui/r17_source_data_contract.py`

Это рабочий stage‑1 scaffold. Он **не** внедряет новую геометрию в runtime сам по себе,
а только фиксирует канон, валидацию и semantic-preserving migration seed для R16.


## R20 geometry source-data (double wishbone hardpoints / cylinder X split)
- `верх_Ц1/Ц2_<перед|зад>_x_относительно_оси_ступицы_м` — продольное смещение верхних шарниров цилиндров относительно оси ступицы.
- `низ_Ц1/Ц2_<перед|зад>_рычаг_крепления` — выбор рычага крепления штока (`верхний_рычаг`/`нижний_рычаг`).
- `низ_Ц1/Ц2_<перед|зад>_ветвь_трапеции` — выбор ветви трапеции (`перед`/`зад`).
- `верхний_рычаг_*_x_относительно_оси_ступицы_м`, `нижний_рычаг_*_x_относительно_оси_ступицы_м` — явные продольные координаты ветвей трапеций для solver/export/animator.


## R31W — visual packaging contract for cylinders/pistons

Для честной 3D-визуализации цилиндров и поршней Animator теперь опирается на
**явные geometry-ключи bundle/export слоя**, а не на скрытые дефолты/эвристики.

Канонические visual geometry keys в `meta_json.geometry`:

- `cyl1_outer_diameter_m`, `cyl2_outer_diameter_m`
- `cyl1_dead_cap_length_m`, `cyl1_dead_rod_length_m`
- `cyl2_dead_cap_length_m`, `cyl2_dead_rod_length_m`

Источник этих ключей в exporter/base contract:

- `cyl*_outer_diameter_m` — из канонических `диаметр_поршня_Ц*` + `стенка_толщина_м`
- `cyl*_dead_*_length_m` — из `мёртвый_объём_камеры` и площадей `диаметр_поршня_Ц*` / `диаметр_штока_Ц*`

Важно:

- это не runtime alias и не скрытая совместимость;
- если этих ключей нет в bundle, Animator обязан предупреждать и не выдумывать 3D packaging.

## R31X — orientation law for cylinder body / rod / piston

Для visual packaging действует жёсткая семантика оси `cyl*_top -> cyl*_bot`:

- `cyl*_top` — frame-side / cap-side крепление цилиндра;
- `cyl*_bot` — arm-side / rod-end крепление штока;
- `положение_штока_*_м` — **выдвижение штока** в диапазоне `0..Stroke`, а не «ход поршня от cap к rod-end».

Следствие для Animator:

- body должен читаться как **цилиндр к раме**;
- rod должен читаться как **шток к рычагу**;
- contract-derived piston plane при росте `stroke_pos` двигается **к cap/frame side**, а не к arm side.

Если consumer визуально получает обратную картину, это считается нарушением visual contract,
а не «допустимой интерпретацией упаковки».



## R31Y — honest fallback until explicit gland/body-end contract exists

Текущий contract уже знает:

- solver-point axis `cyl*_top -> cyl*_bot`
- bore / rod / outer diameter
- dead lengths
- stroke signal (`положение_штока_*_м`)

Но contract **ещё не экспортирует** явную внешнюю точку `gland/body-end` корпуса.
Поэтому Desktop Animator на R31Y использует такой честный fallback:

- `housing_seg = top -> bot` как прозрачная оболочка по всей solver-axis;
- `rod_seg = piston_plane -> bot` как точный шток;
- `piston_plane` как точный contract-derived диск;
- billboard/scatter piston markers — только debug-layer, hidden by default.

Это сделано специально, чтобы не выдумывать внешнюю длину корпуса и не путать пользователя
жёлтыми шарами, которые можно принять за frame mounts.


## R31Z — native live-GL layout + honest cylinder internals

Live 3D окно больше не должно жить через специальный reparent/external-window путь как
основной сценарий пользователя. Для contract/UX действует правило:

- 3D panel использует **native dock/floating** поведение `QDockWidget`;
- если пользователь двигает/ресайзит live GL панель во время playback, Animator обязан
  **временно ставить playback на паузу**, дождаться стабилизации layout и только потом
  возвращать обновление кадров;
- user-facing path не должен снова маскировать crash-path фразой про «безопасное окно».

Для цилиндров/pistons user-facing 3D слой в R31Z обязан показывать не просто «целый цилиндр», а:

- `housing_shell` — честную outer shell по pin-to-pin оси (пока нет explicit `gland/body-end` key);
- `body_chamber` — точную cap-side внутреннюю камеру `top -> piston_plane`;
- `rod` — точный шток `piston_plane -> bot`;
- `piston_plane` / `piston_ring` — точную плоскость поршня без fake thickness и без point-sprite шариков.

Диагностические point/billboard markers не являются частью user-facing contract. Если они вообще
существуют, они должны быть debug-only. Для обычного режима запрещены загадочные «шарики», которые
можно спутать с точками крепления к раме или с внутренними деталями цилиндра.
