# Модуль OED/Identifiability (FIM) — oed_worker_v1_fim.py

Этот модуль — «следующий шаг» после **fit_worker**: сначала проверяем, *какие параметры вообще наблюдаемы* и *какие тесты наиболее информативны*.

Идея:
- выбираем набор параметров `θ` (ваш `fit_ranges_json`),
- выбираем, какие сигналы мы «наблюдаем» (pressures/ходы/углы/расходы и т.д.),
- для каждого теста считаем численную чувствительность `∂y/∂θ` и строим **FIM = JᵀJ**,
- по `logdet(FIM)` (D-opt) оцениваем информативность теста и делаем жадный подбор порядка тестов.

## Что вы получаете на выходе
- `rank` FIM: сколько независимых комбинаций параметров реально наблюдается,
- `corr` матрицу корреляций: какие параметры «слиплись» и будут плохо разделяться при фиттинге,
- **рекомендованный порядок тестов** (жадный D-opt): какие эксперименты добавить в первую очередь.

## Быстрый запуск

Пример: анализируем 8 параметров по 3 сигналам из `df_main`.

```bash
python oed_worker_v1_fim.py \
  --model model_pneumo_v8_energy_audit_vacuum.py \
  --worker opt_worker_v3_margins_energy.py \
  --base_json base.json \
  --fit_ranges_json fit_ranges.json \
  --observables_json observables.json \
  --report_json oed_report.json \
  --progress_json oed_progress.json \
  --sample_stride 5 \
  --record_stride 5 \
  --use_smoothing_defaults
```

### observables.json (пример)

```json
[
  {"model_key": "перемещение_рамы_z_м", "weight": 1.0},
  {"model_key": "крен_phi_рад", "weight": 1.0},
  {"model_key": "давление_ресивер3_Па", "weight": 1e-5}
]
```

Если нужно наблюдать давления узлов/расходы рёбер — используйте префиксы:
- `p:ИмяУзла`
- `q:ИмяРебра`
- `open:ИмяРебра`

Тогда модуль автоматически включит `record_full=True`.

## Как интерпретировать отчёт

- `tests[...].sens_rms`: RMS-чувствительность (по всем наблюдаемым точкам) — быстро показывает «мертвые» параметры.
- `tests[...].logdet`: информативность конкретного теста.
- `suite_total.corr`: корреляции при использовании ВСЕГО набора тестов.
- `greedy_D_opt.order`: порядок добавления тестов (самые информативные — первыми).

## Важно
- Это **практическая** (численная) идентифицируемость вокруг базовой точки `base.json`.
- Если модель негладкая (много `if/clip/max`) — включайте `--use_smoothing_defaults`, иначе производные будут «шуметь».
