# C1-интерполяция табличной пружины (PCHIP)

## Зачем это нужно
В исходной модели (и во многих инженерных прототипах) табличная характеристика пружины
`F(x)` вычислялась через `np.interp`, т.е. *кусочно-линейно*.

Это хорошо для прямого расчёта, но для **градиентной оптимизации** и **автодиффа**
кусочно-линейная интерполяция приводит к разрывам производной `dF/dx` в узлах таблицы.
В результате:

- оптимизаторы могут «прыгать» на узлах,
- градиенты становятся шумными,
- требуется более маленький шаг интегрирования, чтобы сохранять устойчивость.

В релизе **v6.41** добавлен режим **C1-аппроксимации** таблицы пружины через
монотонный кубический интерполятор **PCHIP** (Piecewise Cubic Hermite Interpolating Polynomial).

PCHIP:
- проходит через узлы таблицы,
- имеет непрерывную первую производную (C1),
- сохраняет монотонность для монотонных данных (shape-preserving).

## Как включить
По умолчанию сохраняется поведение базы: `linear`.

В `params` добавьте:

```json
{
  "пружина_интерполяция": "pchip"
}
```

Допустимые значения:
- `linear` (default)
- `pchip` (алиасы: `c1`, `monocubic`, `monotone_cubic`)

Фактически применённый режим записывается в `df_atm`:
- `spring_interp_mode`.

## Важное про физику таблицы
Чтобы сохранить физическую корректность, в модели выполняется нормализация:
- отрицательные `x` и `F` игнорируются (пружина только на сжатие),
- точки сортируются по `x`,
- дубликаты `x` объединяются (берётся максимальная `F`),
- `F(x)` принудительно делается неубывающей через `maximum.accumulate`.

Это защищает от случайных «ломаных» таблиц и делает обратную задачу `x(F)` устойчивой.

## Подбор `x0` согласован с выбранной интерполяцией
Для нулевой позы вычисляется `x0`, такое что:

`F_table(x0) ≈ F_target`

В v6.41 это делается **через единый класс** `SpringForceTable`:
- для `linear` — точная линейная инверсия внутри сегмента,
- для `pchip` — монотонный бисекционный поиск в нужном интервале.

Это уменьшает расхождения между «статикой» (x0) и дальнейшей динамикой.
