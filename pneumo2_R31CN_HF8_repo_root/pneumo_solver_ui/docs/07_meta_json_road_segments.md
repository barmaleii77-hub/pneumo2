# Сценарий «сегменты‑кольцо» (ring_v2)

Эта версия приложения **полностью перешла** на новый редактор сценариев «сегменты‑кольцо».

- Старый редактор сценариев **удалён**.
- Совместимость со старыми сценариями **не обеспечивается**.
- Все параметры следуют **ABSOLUTE LAW** (см. `00_READ_FIRST__ABSOLUTE_LAW.md`).
  - Никаких «алиасов» (`speed_kmh`, `v0_kmh`, `v_end_kmh`, …) — **запрещено**.
  - Никаких «выдуманных параметров».
  - Всё, что вычисляется автоматически, должно быть явно помечено как производное/служебное и не должно становиться «новым входным параметром».

## Что такое ring_v2

`ring_v2` — это сценарий, заданный **последовательностью сегментов**, которые образуют **замкнутую кольцевую трассу**.

Сценарий хранится в одном JSON-файле и используется для генерации двух входных CSV:

- `scenario_<TAG>_spec.json` — исходный сценарий (единственный источник истины)
- `scenario_<TAG>_road.csv` — профиль дороги (левый/правый след)
- `scenario_<TAG>_axay.csv` — продольное/поперечное ускорение

По умолчанию редактор сохраняет файлы в `workspace/scenarios/`.

## Каноническая схема `scenario_json` (ring_v2)

### Верхний уровень

```json
{
  "schema_version": "ring_v2",
  "title": "…",

  "v0_kph": 20.0,
  "segments": [ ... ]
}
```

Поддерживаемые (канонические) поля верхнего уровня:

- `schema_version`: строка, строго `"ring_v2"`
- `title`: строка (необязательно)
- `v0_kph`: начальная скорость, км/ч
- `segments`: список сегментов (см. ниже)

Поля, которые задаются редактором/генератором и считаются **техническими настройками генерации**:

- `dt_s`: шаг времени (сек)
- `n_laps`: число повторов набора сегментов по времени
- `wheelbase_m`: колёсная база для построения заднего следа (**служебная производная** от канонического параметра модели `база`)
- `dx_m`: шаг дискретизации по расстоянию для дороги
- `seed`: seed для генератора

> Эти поля не являются «физическими параметрами модели». Они относятся к генерации входных сигналов.

### Сегмент

```json
{
  "drive_mode": "STRAIGHT",
  "duration_s": 10.0,

  "speed_kph": 20.0,

  "turn_radius_m": 60.0,

  "road": { ... },
  "events": [ ... ]
}
```

Канонические поля сегмента:

- `drive_mode`: режим движения, одно из:
  - `STRAIGHT` — прямолинейное движение
  - `TURN_LEFT` / `TURN_RIGHT` — поворот (поперечное ускорение рассчитывается из `turn_radius_m`)
  - `ACCEL` / `BRAKE` — разгон/торможение (линейное изменение скорости)
- `duration_s`: длительность сегмента, сек

Поля скорости:

- `speed_kph` — **только** для `STRAIGHT`, `TURN_LEFT`, `TURN_RIGHT`
- `v_end_kph` — **только** для `ACCEL`, `BRAKE`

Параметры поворота:

- `turn_radius_m` — радиус поворота (м). Используется в `TURN_*`.

### `road`: параметры дороги на сегменте

`road.mode` определяет тип дороги на сегменте.

#### ISO 8608 (стохастический профиль)

```json
"road": {
  "mode": "ISO8608",
  "iso_class": "C",
  "gd_pick": "mid",
  "waviness_w": 2.0,
  "seed": 12345,
  "left_right_coherence": 0.6,
  "gd_n0_scale": 0.25
}
```

Канонические поля:

- `mode`: `"ISO8608"`
- `iso_class`: класс ISO, `A`…`H`
- `gd_pick`: уровень внутри класса, одно из `lower` / `mid` / `upper`
- `waviness_w`: показатель степенного закона PSD (инженерный дефолт обычно `2.0`)
- `seed`: локальный seed сегмента для воспроизводимости профиля
- `left_right_coherence`: корреляция левого/правого следа (0…1)
- `gd_n0_scale`: масштаб Gd(n0) (безразмерно)

`left_right_coherence` реализован как **коэффициент смеси** между независимым правым следом и левым следом:

- `0.0` → левый и правый треки независимы
- `1.0` → правый трек совпадает с левым

⚠️ `road_gd_n0_target_m3` в `ring_v2` **не поддерживается**. Для управления шероховатостью используйте только каноническую пару `gd_pick` + `gd_n0_scale`.

#### SINE (синусоидальная дорога)

```json
"road": {
  "mode": "SINE",
  "aL_mm": 5.0,
  "aR_mm": 5.0,
  "lambdaL_m": 2.0,
  "lambdaR_m": 2.0,
  "phaseL_deg": 0.0,
  "phaseR_deg": 0.0
}
```

Канонические поля:

- `mode`: `"SINE"`
- `aL_mm`, `aR_mm`: амплитуды (мм)
- `lambdaL_m`, `lambdaR_m`: длины волн (м)
- `phaseL_deg`, `phaseR_deg`: фазы (градусы)

Также поддерживаются поля рандомизации (все — **канонические**):

- `rand_aL_lo_mm`, `rand_aL_hi_mm`
- `rand_aR_lo_mm`, `rand_aR_hi_mm`
- `rand_lambdaL_lo_m`, `rand_lambdaL_hi_m`
- `rand_lambdaR_lo_m`, `rand_lambdaR_hi_m`
- `rand_phaseL_lo_deg`, `rand_phaseL_hi_deg`
- `rand_phaseR_lo_deg`, `rand_phaseR_hi_deg`

### `events`: локальные дефекты

```json
"events": [
  {
    "kind": "pothole",
    "side": "both",
    "start_m": 8.0,
    "length_m": 0.8,
    "depth_mm": 10.0,
    "ramp_m": 0.2
  }
]
```

Канонические поля события:

- `kind`: строка (описательная метка; на генерацию не влияет)
- `side`: `left` / `right` / `both`
- `start_m`: начало дефекта внутри сегмента (м)
- `length_m`: длина дефекта (м)
- `depth_mm`: глубина/высота дефекта (мм). **Единственный допустимый ключ глубины.**
- `ramp_m`: длина «скоса» (м), опционально

## Почему нет алиасов

Это принципиальное правило проекта:

- если параметр называется `speed_kph`, то **только так**
- если параметр называется `depth_mm`, то **только так**

Если в JSON встречаются старые имена (`speed_kmh`, `depth_m`, …), они считаются **ошибкой данных**. Приложение должно:

- не падать,
- предупредить,
- и использовать значения по умолчанию/безопасные значения.
