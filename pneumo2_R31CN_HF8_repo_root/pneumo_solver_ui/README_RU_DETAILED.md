## 1) Что это и что умеет

Проект решает практическую задачу: **посмотреть “что происходит” в пневмосистеме и механике** во времени по результатам симуляции,
а затем — оптимизировать параметры и сравнивать решения.

### Ключевые функции UI

- **Единый таймлайн (playhead)**: один ползунок времени управляет:
  - анимацией **пневмосхемы**,
  - анимацией **механики**,
  - вертикальной линией времени на **графиках** + маркерами значений.
- **Кликабельность**:
  - клик по ветке/узлу на схеме → выбирает это в графиках и в анимации,
  - клик по графику Plotly → выбирает соответствующую кривую и (опционально) прыгает по времени.
- **События/алёрты**:
  - автодетекция “колесо в воздухе”, “шток у упора”, “вакуум/перепрессовка”, “дребезг клапанов” и т.п.
  - метки событий на таймлайне и на графиках.
- **SVG autotrace (beta)**:
  - автоматически разбирает SVG из CAD/Illustrator: линии, тексты, склейку узлов, граф соединений, полилинии,
  - пытается автоматически построить **mapping JSON** для анимации (ветки/узлы).

## 4) Как устроены данные симуляции

### 4.1 Основные датафреймы

- `df_main` — “главные” временные ряды (углы, штоки, силы шин, иногда “сводные” давления).
- `df_p` — давления в узлах сети (много колонок; используется для “узлов давления” на схеме).
- `df_mdot` — расходы по веткам (используется для анимации потоков и графиков).
- `df_open` — состояния открытия элементов (для “дребезга” и визуализации on/off).

### 4.2 Запуск “полного лога”
Чтобы `df_p/df_mdot/df_open` были доступны — нужен режим `record_full=True` (вызывается из UI в “детальном прогоне”).

### 5.3 Graph Studio: произвольные сигналы (v7.32)

Добавлен **конструктор графиков** для быстрых экспериментов без правки кода:

- выбор источника данных: `df_main / df_p / df_mdot / df_open`,
- фильтр сигналов (подстрока или regex),
- пресеты (штоки, колёса+дорога, давления, крен/тангаж),
- 2 режима:
  - **Осциллограф (stack):** каждый сигнал своей строкой (Plotly subplots, общий X),
  - **Overlay:** несколько сигналов в одном графике (одна ось Y),
- **клик по графику → прыжок playhead по времени** (через `st.plotly_chart(..., on_select="rerun")`),
- decimation:
  - `minmax` (сохраняет пики),
  - `stride` (быстро),
- renderer:
  - `svg` (без WebGL‑контекстов),
  - `webgl` (Scattergl, быстрее на больших данных),
- `hovermode="x unified"` + `hoversubplots="axis"` для “общего курсора” по всем подграфикам,
- экспорт выбранных сигналов в **CSV** и **Excel**,
- быстрые статистики min/max/mean на выбранном временном окне.

**Зачем это нужно:** на манёврах (поворот/слалом) удобно одновременно держать на экране:
- 4 gap/контактные параметры,
- 4 перемещения колёс + 4 профиля дороги,
- крен/тангаж/рысканье,
- силы шин и ходы штоков.

## 7) Пневмосхема (SVG): ручная разметка mapping JSON

### 7.1 Формат mapping JSON (v2)
```json
{
  "version": 2,
  "viewBox": "0 0 1920 1080",
  "edges": {
    "edge_name_from_df_mdot": [
      [[x,y],[x,y],[x,y]]
    ]
  },
  "nodes": {
    "node_name_from_df_p": [x,y]
  },
  "meta": { "generated_by": "...", "params": {...} },
  "edges_meta": { "...": {...} },
  "nodes_meta": { "...": {...} }
}
```

- `edges[...]` — список полилиний (может быть несколько сегментов на одну ветку)
- `nodes[...]` — координаты подписи давления на схеме

## 8) Авторазметка из SVG (autotrace) — что это и как пользоваться

Файл `svg_autotrace.py` делает:

1) парсинг SVG:
   - `<line x1 y1 x2 y2>` — геометрия линий (труб),
   - `<text ... transform="matrix(a b c d e f)">` — позиции подписей,
2) склейку концов линий в узлы графа (tol_merge),
3) построение графа соединений,
4) разложение графа на “полилинии” между junction (deg≠2),
5) сопоставление текстовых меток с ближайшими полилиниями,
6) (опционально) упрощение полилиний (RDP epsilon),
7) сбор mapping JSON “черновика”.

### Важные параметры
- `tol_merge` — расстояние (px) для склейки концов линий в один узел.
- `max_label_dist` — насколько далеко текст может быть от трубы, чтобы мы считали, что это “про неё”.
- `min_name_score` — порог fuzzy‑сопоставления имён.
- `simplify_epsilon` — упрощение полилиний (уменьшает размер mapping).
- `node_snap_max_dist` — max dist для snap к junction.


### 8.2) Путь по схеме (connectivity / shortest path)

Начиная с **v7.15** добавлен режим *“Путь по схеме”*:

- выбираешь **две текстовые метки** (старт / финиш),
- UI “врезает” эти точки в ближайшие сегменты графа труб (split edge),
- запускает **Dijkstra** по взвешенному графу (вес = длина сегмента),
- подсвечивает найденный маршрут поверх SVG (оранжевая пунктирная линия + маркеры концов),
- можно скачать `svg_route.json` (включает точки маршрута и диагностические поля attach_start/attach_end).

Начиная с **v7.16** найденный маршрут можно **записывать прямо в `mapping.edges[ветка]`** (режим: *заменить* или *добавить сегмент*).

Начиная с **v7.17** можно выбирать START/END **кликом по текстовой подписи** на схеме (без длинных списков).

Начиная с **v7.18** добавлено ещё ускорение разметки:

- **Shift+клик** по схеме → выбрать **START**  
- **Ctrl/Cmd+клик** по схеме → выбрать **END**  
- можно кликать **не точно по тексту**, а **рядом** — будет выбран ближайший текст в заданном радиусе (*“Радиус поиска ближайшей метки”*).
- добавлен “мини‑визард” покрытия `mapping.edges`: видно сколько веток размечено/не размечено, есть кнопка **“Следующая неразмеченная”** и опция **автоперехода** после записи маршрута.


Начиная с **v7.19** добавлен **ассистент разметки веток (guided)** прямо в блоке «Путь по схеме»:

- целевая ветка `df_mdot` выбирается **до** построения пути (и используется при записи маршрута в `mapping.edges`),
- показано покрытие `mapping.edges` + «следующая неразмеченная» (теперь доступно сразу, ещё до маршрута),
- кнопка **«Автофильтр по имени ветки»** — быстро сужает список меток по сигнатуре (ЛП1/ПЗ2/… если найдена),
- **подсказки START/END**: список top‑K текстовых меток SVG, похожих на имя ветки (fuzzy‑score), с кнопками **START/END**,
- кнопка **«↔ Поменять START/END»**,
- опция **«Очистить маршрут после записи»** — удобно для пакетной разметки, чтобы старый маршрут не мешал следующей ветке.






Начиная с **v7.20** добавлен **AUTO‑конвейер разметки `mapping.edges`** внутри блока «Путь по схеме»:

- **AUTO: текущая** — по текущей ветке `df_mdot` автоматически:
  1) подбирает кандидатов START/END (top‑K по fuzzy‑score),
  2) выбирает пару по стратегии (*Top2 / Best+Farthest / FarthestPair*),
  3) строит маршрут (Dijkstra по графу труб),
  4) записывает маршрут в `mapping.edges[ветка]`,
  5) (опционально) готовит переход на следующую неразмеченную ветку.
- **AUTO: batch** — пробует размечать **N неразмеченных** веток подряд и выдаёт таблицу + CSV отчёт (успех/ошибка, метки, score, длина, точки).
- В `mapping.edges_meta[ветка]` сохраняется “почему так получилось”: какие метки использованы, их score, стратегия, длина маршрута и timestamp.
- Для защиты от “маршрут через всю схему” есть лимит **MaxLen (px)**: если маршрут длиннее — запись не делается (ошибка в отчёте).

Практический совет: перед AUTO часто полезно нажать **«Автофильтр по имени ветки»** (из guided‑блока),
чтобы список меток (и кандидаты) соответствовали именно нужной зоне схемы.

Это полезно для:
- проверки, что “подпись” действительно рядом с нужной трубой,
- поиска разрывов/несвязностей в SVG,
- быстрой навигации по сложной схеме.

Ограничения:
- граф строится только по `<line>`; если в исходном SVG есть `<path>`/кривые — их нужно предварительно конвертировать в линии (или расширить парсер),
- если на схеме много дублей текстов (например много раз встречается “Ресивер”), выбирай нужный экземпляр по координатам в списке.


- `prefer_junction` — если рядом есть junction, выбираем его.

### 8.1) Путь по схеме (connectivity) → записать в mapping.edges (v7.16)

Инструмент **«Путь по схеме (connectivity beta)»** строит маршрут по извлечённому графу труб **между двумя текстовыми метками** на SVG
(например: «Ресивер 2» → «Рег.»). Маршрут отображается поверх схемы и может быть выгружен как `svg_route.json`.

**Новое в v7.16:** найденный маршрут можно **сразу записать в `mapping.edges[<ветка модели>]`** (semi‑auto разметка веток):

1) Построй путь в блоке **«Путь по схеме (connectivity beta)»** и убедись, что он корректный.
2) В секции **«Привязать найденный путь к ветке модели (mapping.edges)»** выбери целевую ветку из `df_mdot`.
3) Выбери режим:
   - **Заменить** — заменить маршрутом все сегменты `mapping.edges[ветка]`.
   - **Добавить сегмент** — добавить маршрут как ещё один сегмент (когда ветка на схеме рисуется несколькими кусками).
4) Нажми **«Записать маршрут»**.
5) Прокрути вниз к **«Анимация по схеме (по mapping JSON)»** и **скачай mapping JSON** в файл (для фиксации).

Это резко ускоряет сбор `mapping.edges`: вместо ручного рисования полилиний достаточно выбрать две подписи (start/end)
и назначить результат конкретной ветке модели.

**Ограничения/заметки:**
- маршрут ищется по геометрии `<line>` (если трубки в SVG представлены иначе — парсер надо расширять),
- подписи `P/Q/PQ` часто являются “шумом” — используйте фильтр меток.


## 9) Snapping узлов давления (важное улучшение v7.14)

Проблема: текстовая метка может быть около середины длинного сегмента, а ближайший node‑узел графа будет далеко.

Решение v7.14:
- если рядом есть **junction (deg≠2)** → snap в junction,
- иначе → snap в **ближайшую точку на полилинии** (проекция на сегмент).

Это стабилизирует координаты узлов давления на схеме.

### 8.3) Quality/Review разметки (edges_meta.review) — NEW v7.21

В v7.21 добавлен **слой контроля качества и ревью** для авто/ручной разметки маршрутов.

**Что нового:**
- `svg_autotrace.evaluate_route_quality()` считает метрики маршрута: длина, detour (извилистость), число точек, грубое число поворотов, самопересечения (для умеренных размеров), расстояние метка→трубка (если есть attach_*).
- UI сохраняет качество в `mapping.edges_meta[EDGE].quality` и статус ревью в `mapping.edges_meta[EDGE].review.status`.
- В блоке **«Анимация по схеме (по mapping JSON)»** появился expander **Review / Quality**:
  - **Recompute quality (all)** — пересчитать метрики для всех веток, где есть геометрия.
  - **Approve all PASS** — автоматически проставить `approved` всем веткам с `quality.grade=PASS`.
  - Таблица с фильтрами по status/grade и экспортом CSV.
  - Ручное изменение статуса (`approved/pending/rejected`) и заметки.
  - **Clear geometry** — удалить `mapping.edges[edge]` и пометить ветку как `rejected` (meta остаётся для истории).

**Как пользоваться на практике (рекомендованный поток):**
1) Сделайте AUTO batch на N веток.
2) Откройте **Review / Quality**, нажмите **Recompute quality (all)**.
3) Нажмите **Approve all PASS** (получите быстрый «черновой approved»).
4) Пройдитесь по `pending`/`WARN` веткам вручную: выберите edge → посмотрите на схеме маршрут/анимацию → исправьте или отклоните.
5) В анимации можно включить фильтр **Только APPROVED**.

⚠️ Это эвристики. PASS не гарантирует 100% правильность, но помогает быстро отсеивать откровенно плохие маршруты.


### 8.4) Review overlay на SVG + быстрые approve/reject — NEW v7.22

В v7.22 добавлена **цветовая “накладка” статусов ревью прямо поверх SVG‑схемы**.

**Что это даёт:**
- Вы сразу видите, какие ветки размечены и утверждены (`approved`), какие требуют проверки (`pending`), какие отклонены (`rejected`).
- Можно менять статус **прямо кликом по линии** (в режиме Review hotkeys).

**Где включать:**
- `Анимация → Пневмосхема (SVG) → Анимация по схеме (по mapping JSON)`
  - чекбокс **“Показать review overlay”**
  - чекбокс **“Review hotkeys”**
  - фильтр **“Показывать статусы”**

**Горячие действия (работают, когда включён Review hotkeys):**
- `Shift + клик по линии overlay` → `approved`
- `Ctrl/Cmd + клик` → `rejected`
- `Alt + клик` → `pending`
- `обычный клик` → просто выбирает edge (синхронизация с графиками/анимацией)

**Важно:** overlay рисуется из `mapping.edges`, а статусы берутся из `mapping.edges_meta[edge].review.status`.

Технически кликабельность обеспечена тем, что у SVG‑путей включены pointer events на stroke (`pointer-events: stroke`).


### 8.5) Review HUD + pending conveyor на схеме — NEW v7.23

В v7.23 добавлены инструменты для **быстрого “ревью‑конвейера”** прямо поверх SVG‑схемы.

#### Что появилось

1) **HUD‑панель Review на схеме**  
Показывается прямо поверх SVG (слева), если включено:
- `Показать review overlay`
- `HUD на схеме`

HUD показывает:
- счётчики `approved / pending / rejected / unknown / total` (по `mapping.edges`)
- текущую выбранную ветку `sel` и её `status`
- кнопки навигации:
  - **◀ pending** — перейти к предыдущей ветке со статусом `pending/unknown`
  - **pending ▶** — перейти к следующей
- кнопку **Only pending / Show all** — переключает фильтр статусов overlay:
  - pending‑only режим: `["pending", "unknown"]`
  - полный режим: `["approved","pending","rejected"]`

2) **Review conveyor (pending-first) в Streamlit‑UI**  
В блоке:
`Анимация → Пневмосхема (SVG) → Анимация по схеме (по mapping JSON)`
появился expander **Review conveyor (pending-first)**:
- метрики/счётчики статусов (по `mapping.edges_meta[edge].review.status`)
- кнопки `◀ Prev pending` / `Next pending ▶` (то же самое, но в UI справа)
- чекбокс **Auto-advance после approve/reject** — после `Shift`/`Ctrl` клика по линии overlay автоматически выбирается следующая pending/unknown ветка

#### Как пользоваться (рекомендованный “конвейер”)

1) Сделайте AUTO batch разметку.
2) Откройте `Review / Quality` (v7.21) → `Recompute quality (all)` → `Approve all PASS`.
3) Включите `Показать review overlay` + `HUD на схеме` + (по желанию) `Review hotkeys`.
4) Пройдитесь по pending/unknown:
   - `pending ▶` листает следующую ветку
   - `Shift + клик` по линии → approve, auto‑advance на следующую (если включено)
   - `Ctrl/Cmd + клик` → reject, auto‑advance

#### Техническая реализация (для понимания)

- HUD и overlay рисуются **во фронтенде компонента** `components/pneumo_svg_flow/index.html`.
- Нажатия кнопок HUD отправляют в Python события через `streamlit:setComponentValue`:
  - `{kind:'review_nav', action:'next_pending'|'prev_pending'}`  
  - `{kind:'review_filter', mode:'toggle_pending_only'}`
- Python‑часть (`pneumo_ui_app.py`) обрабатывает эти события в `consume_svg_pick_event()` и:
  - меняет `st.session_state["svg_selected_edge"]` (чтобы подсветить ветку)
  - меняет `st.session_state["svg_review_statuses"]` (чтобы отфильтровать overlay)
  - при review_action может делать auto‑advance

## 10) Механика: анимация и клики

В блоке **«Механика»** теперь два таба:

### 10.1 2D (схема)

2D‑компонент показывает:
- фронтальный вид (крен) + боковой (тангаж),
- условные колёса и раму,
- ходы штоков.

Клики по колёсам/осям:
- обновляют выбранные “углы/оси” в UI,
- графики механики перестраиваются по выбранному.

### 10.2 3D (машинка wireframe) — v7.24…v7.26

3D‑таб даёт **пространственную** картинку, которую можно вращать мышью (orbit‑камера):

- рама — параллелепипед (wireframe),
- 4 колеса — цилиндры (wireframe) + «спица» (видно вращение),
- профиль дороги под каждым колесом — кривая линия,
- траектория (центр кузова) — линия/«след».

Управление камерой:
- ЛКМ drag — вращение (az/el),
- колесо мыши — zoom,
- пресеты: ISO / TOP / FRONT / SIDE,
- опции: «Камера следует за машиной» и «Камера поворачивается по yaw».

Скорость/время:
- движение идёт от **общего playhead** (localStorage),
- при `speed = 1×` 1 сек симуляции ≈ 1 сек реального времени (рендер через `requestAnimationFrame`).

Манёвры (как строится траектория X/Z и yaw):
- **Прямая**: `x = v0 * t`, `z = 0`.
- **Слалом**: `z = A * sin(2πt/T)`, yaw вычисляется по направлению скорости.
- **По ax/ay**: интегрируем ускорения → скорости → координаты; yaw берём по направлению скорости.

Новое в **v7.26** (важно для «реального ощущения»):
- Дорога теперь рисуется в тех же координатах, что и колёса/рама: применяются `y_sign`, `road_y_offset` и опция `road - R` (если `road` в df = уровень центра колеса).
- Режим дороги:
  - `track` — «след по траектории» (используем `yaw_i` и `path x/z` на каждом сэмпле),
  - `local` — «под машиной» (окно вокруг текущего момента рисуется в текущей системе кузова; удобно для калибровки контакта).
- Реалистичнее вращение колёс (`spin_per_wheel`): считаем путь **каждого** колеса. В повороте внутренние/наружные колёса крутятся по‑разному.
- Индикатор контакта колеса с дорогой (`show_contact`): вертикальный маркер gap/penetration + крестик на road.
- Стойки/подвеска (`show_suspension`): линии от углов рамы к центрам колёс.

Где код:
- Python: `pneumo_ui_app.py` (подготовка массивов и UI‑параметров),
- JS/Canvas: `components/mech_car3d/index.html`.

### 10.3) Механика 3D (wireframe) — обновление v7.27

В v7.27 прокачана 3D‑анимация «машинка + дорога» так, чтобы манёвры (повороты/слалом) были визуально очевидны и удобны для ревью.

### 10.4) Кликабельность в 3D (выбор колёс/осей) + синхронизация графиков — v7.28

В v7.28 3D‑вид «машинка wireframe» стал **кликабельным**:

- **Клик по колесу** выбирает угол подвески (`ЛП/ПП/ЛЗ/ПЗ`).
- **Клик в центре оси** (между левым и правым колесом) выбирает **перед/зад** (`front/rear`) и сразу выделяет оба угла оси.
- Выделение **синхронизируется** с механическими графиками (через `mech_selected_corners` / `mech_plot_corners`).

**Как это устроено (важно для доработок):**
- 2D‑мех‑компонент пишет события в `st.session_state["mech2d_pick_event"]`.
- 3D‑мех‑компонент пишет события в `st.session_state["mech3d_pick_event"]`.
- `consume_mech_pick_event()` берёт **самое новое** событие по `ts` и применяет его к общей модели выбора.
- В 3D выделенные колёса **подсвечиваются** (толще/ярче ring/spoke), чтобы сразу было видно что выбрано.

> Примечание: если клик «не берётся», значит это был drag (мы специально отсекаем клики при заметном перемещении мыши, чтобы не ловить ложные срабатывания во время вращения/панорамирования).

### 10.5) Механика 3D — обновление v7.29 (hover + follow + minimap)

В v7.29 добавлены функции, которые сильно ускоряют «отладку манёвров» (повороты/слалом) прямо по анимации:

**1) Hover‑подсказки (без перерисовки Streamlit)**
- Наведи курсор на колесо или центр оси → появляется подсказка:
  - `wheel_y` (высота центра колеса),
  - `road_y` (высота поверхности дороги под этим колесом),
  - `gap` (зазор: (wheel_y - R) - road_y),
  - `v` (скорость), `t` (время).
- Подсказка обновляется **каждый кадр**, т.е. значения «едут» вместе с playhead.
- Hover реализован полностью внутри HTML/JS компонента, поэтому **не вызывает rerun Streamlit**.

**2) Камера следует за выбранным колесом/осью**

  то цель (target) орбит‑камеры переносится в центр выбранного колеса (или середину оси).
- Удобно ловить проблемы контакта/хода подвески в конкретном углу.

**3) Мини‑карта (вид сверху)**
- Опциональная “TOP MAP” поверх сцены (в single‑view режиме).
- Показывает траекторию кузова за последние ~600 точек и направление yaw.

Параметры в UI (Механика → 3D):
- Hover‑подсказки (wheel/road/gap)
- Камера следует за выбранным колесом/осью
- Мини‑карта (вид сверху) + размер (px)


### 10.4) Механика 3D — gap heat / min‑gap / smooth follow (v7.30)

В v7.30 добавлены инструменты, которые делают 3D‑анимацию **и нагляднее, и полезнее для диагностики контакта колеса с дорогой** (особенно на манёврах: поворот/слалом).

**Новые возможности:**
- **Цвет по gap (контакт/зазор):** колёса и маркеры контакта окрашиваются по величине `gap = (wheel_bottom - road_surface)`:
  - отрицательный gap → проникновение (красный),
  - около нуля → контакт (зелёный),
  - положительный gap → зазор (жёлтый).
- **HUD: gap и min‑gap:** вывод текущего gap по всем 4 колёсам + минимального gap за окно N точек назад.
- **Окно min‑gap + шаг анализа:** чтобы не тормозить отрисовку, min‑gap считается с прореживанием.
- **Маркер контакта при hover:** при наведении на колесо/ось контакт рисуется **крупнее и толще** (цветом по gap).
- **Сглаживание target камеры (follow smoothing):** чтобы камера не «дрожала» на шумном профиле дороги при follow‑режимах.

**Где включать в UI:**
Механика → 3D (машинка wireframe):
- «Цвет по gap …»
- «Шкала gap (м) для цвета»
- «Показывать gap/min‑gap в HUD»
- «Окно min‑gap …» + «Шаг анализа …»
- «Маркер контакта при hover …»
- «Сглаживание target …»



### 10.5) Экспорт анимации: PNG + запись WebM (v7.31)

В v7.31 добавлены **кнопки экспорта прямо в HUD 3D‑вида** (без Python‑обработки, работает полностью в браузере):

- **PNG** — сохранить текущий кадр (`canvas.toDataURL("image/png")`).
- **REC/STOP** — запись видео в **WebM** через `canvas.captureStream(fps)` + `MediaRecorder`.
  - FPS выбирается в выпадающем списке (15/30/60).
  - После STOP браузер автоматически скачивает файл `mech3d_YYYY-MM-DDTHH-MM-SS.webm`.

**Ограничения/заметки:**
- Запись зависит от поддержки браузером API `MediaRecorder` и `HTMLCanvasElement.captureStream()` (обычно хорошо работает в Chrome/Edge).
- Запись фиксирует **реальный рендер** того, что на canvas (включая мультивид, треки, gap‑heat и т.п.).
- Битрейт/кодек выбирает браузер (в коде есть попытка выбрать VP9→VP8→WebM).


## 11) Производительность и стабильность

### Почему анимация плавная
SVG и механика читают playhead из `localStorage` и не требуют rerun каждую миллисекунду.

### Почему графики обновляются ступенчато
Графики живут в Python, и обновление playhead‑линии происходит с частотой `send_hz` (throttle).
Если сделать `send_hz` слишком высоким — Streamlit начнёт часто rerun и станет тяжело.

## 13) Частые проблемы и диагностика

1) “UI падает при клике / selection”
- Проверь, что Plotly установлен.
- Проверь, что `st.plotly_chart` вызван с `on_select="rerun"`.

2) “Streamlit ругается на session_state”
- Любые изменения `st.session_state[key]` для виджета делай ДО создания виджета.
- Поэтому используем `consume_*()` в самом начале run.

3) “Авторазметка промахивается”
- Подними `max_label_dist`, подстрой `tol_merge`.
- Проверь, что SVG не “в кривых transform” (если очень много групповых трансформов — нужно расширять парсер).

4) “В начале симуляции есть рывок/удар, хотя дорога ровная”
- Это обычно означает, что t=0 **не является статическим равновесием** (ускорения не нулевые).
- В моделях **v9_camozzi**, **v9_worldroad** и **v8_patched_smooth_all** добавлен механизм **Static trim t=0**:
  - сначала вычисляется RHS в t=0 и оцениваются ускорения `z̈, φ̈, θ̈, z̈w`;
  - если они превышают пороги, выполняется небольшой нелинейный least-squares (SciPy) по 7 переменным
    (z, φ, θ, z_wheels) с мягкими штрафами на `delta_w≈0` и `φ,θ≈0`.
  - результаты сохраняются в `df_atm` (поля `static_trim_*`).
- Управление параметрами (в `default_base.json`):
  - `static_trim_enable` (bool)
  - `static_trim_tol_accel_m_s2`, `static_trim_tol_accel_rad_s2`
  - `static_trim_max_nfev`
  - `static_trim_bound_z_m`, `static_trim_bound_angle_rad`, `static_trim_bound_zw_m`
  - `static_trim_scale_accel`, `static_trim_scale_delta_m`, `static_trim_scale_angle_rad`
  - `static_trim_force` (всегда пытаться тримовать)

Если ты прогоняешь массовую оптимизацию и хочешь максимум скорости — можно временно выключить `static_trim_enable`.

## Быстрый выбор START/END меток кликом на схеме (NEW v7.17)

Раньше путь по схеме (connectivity) приходилось выбирать через длинные выпадающие списки текстовых меток.
Теперь можно **напрямую кликать по подписи на SVG**:

1) В блоке **«Путь по схеме (connectivity beta)»** нажми:

5) (Если нужно) нажми **«Записать маршрут в mapping.edges»** и выбери ветку df_mdot.

Технически это сделано так:
- SVG‑компонент в режиме `label_pick_mode` подписывается на клики по `<text>` и отправляет в Python событие:
  `{kind:'label', mode:'start|end', ti:<index>, name:<text>, x:<cx>, y:<cy>, ts:<ms>}`
- UI принимает событие ранним `consume_svg_pick_event()` и применяет его до создания виджетов.
