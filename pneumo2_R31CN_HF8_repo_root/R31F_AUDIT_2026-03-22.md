# Аудит R31F — amplitude semantics + baseline→detail recalc

Дата: 2026-03-22
База проверки: `work_r31f` (поверх R31E)

## 1) Что подтверждено по коду

### 1.1. Семантика синуса
В `pneumo_solver_ui/scenario_ring.py` синус строится формулой:

`z(x) = A * sin(...)`

То есть `A` — это именно **амплитуда / полуразмах**, а не полный перепад.

Проверка по коду:
- `scenario_ring.py:139`

Следствие:
- `A = 100 мм` означает диапазон **от -100 до +100 мм** относительно локального нуля,
- полный размах `p-p = 200 мм`.

### 1.2. Почему у пользователя было ощущение «не считает детальный лог»
В `pneumo_solver_ui/app.py` авто-detail после baseline действительно был реализован через `baseline_just_ran`,
но до запуска пересчёта код пытался загрузить detail с диска.

Следствие:
- UI мог показать, что detail «подхватился»,
- но фактически это мог быть **старый cache hit**, а не новый расчёт.

## 2) Что изменено

### 2.1. Ring UI
В `pneumo_solver_ui/ui_scenario_ring.py`:
- поля теперь подписаны как `Амплитуда A (полуразмах), мм`;
- под полями выводится производный `p-p (из A)`;
- добавлено прямое объяснение:
  - `A=100 мм -> -100..+100 мм`,
  - `p-p = 200 мм`.

Проверка по коду:
- `ui_scenario_ring.py:434`
- `ui_scenario_ring.py:471`

### 2.2. Baseline → auto detail
Добавлен отдельный policy helper:
- `pneumo_solver_ui/detail_autorun_policy.py`

В `pneumo_solver_ui/app.py`:
- после свежего baseline текущий `cache_key` помечается как **force-fresh**;
- disk cache для этого key один раз пропускается;
- detail пересчитывается заново;
- источник detail-лога теперь явно показывается в UI.

Проверка по коду:
- `app.py:6287`
- `app.py:6288`
